01 — Design Principles
設計原則與架構說明
從靜態 JSON 檔案遷移至 Firebase Firestore，集中管理所有遊戲資料，解決複雜關聯性與維護困難的問題。
🔗

關聯管理

模組 ↔ 機甲、技能 ↔ 機師、BUFF ↔ 技能等跨實體關聯，透過 Firestore Document ID 引用，取代 JSON 中的字串 ID 難以驗證的問題。

📐

型別枚舉化

模組槽位、職業、裝甲類型、稀有度等分類欄位，統一定義為 TypeScript Enum，前端與後端共用同一份定義，避免拼寫錯誤與不一致。

🛡️

安全規則分層

遊戲資料 Collections：所有人可讀、僅管理員可寫。用戶 Collection：僅本人可讀寫。Firebase Security Rules 強制執行。

⚡

查詢優化

針對常用篩選場景（依職業篩機師、依槽位篩模組、依機甲篩副模組）預設複合索引，確保大資料量下的查詢效能。

🔄

可擴充設計

BUFF Collection 初始僅含名稱與描述，欄位設計為可向後兼容擴充。技能 Collection 預留 tags、relatedEntities 等泛化欄位。

📋

Admin 管理介面

遊戲資料改由網站管理後台（AdminPage）直接讀寫 Firestore，取代 Google Sheets + GitHub Actions 的間接流程。

05 — Security Rules
Firestore 安全規則
遊戲資料 Collection 公開可讀、管理員可寫；用戶資料僅本人可讀寫。
firestore.rulesrules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // ── 輔助函數 ────────────────────────────── function isAuthenticated() { return request.auth != null; } function isOwner(userId) { return isAuthenticated() && request.auth.uid == userId; } function isAdmin() { return isAuthenticated() && get(/databases/$(database)/documents/users/$(request.auth.uid)/profile) .data.role == 'ADMIN'; } // ── 遊戲資料 Collections（所有人可讀，管理員可寫）── match /pilots/{docId} { allow read: if true; allow write: if isAdmin(); } match /mechs/{docId} { allow read: if true; allow write: if isAdmin(); } match /modules/{docId} { allow read: if true; allow write: if isAdmin(); } match /weapons/{docId} { allow read: if true; allow write: if isAdmin(); } match /backpacks/{docId} { allow read: if true; allow write: if isAdmin(); } match /components/{docId} { allow read: if true; allow write: if isAdmin(); } match /skills/{docId} { allow read: if true; allow write: if isAdmin(); } match /buffs/{docId} { allow read: if true; allow write: if isAdmin(); } match /pilotResearch/{docId} { allow read: if true; allow write: if isAdmin(); } match /globalResearch/{docId} { allow read: if true; allow write: if isAdmin(); } match /enemies/{docId} { allow read: if true; allow write: if isAdmin(); } // ── 用戶資料（本人可讀寫）──────────────────── match /users/{userId} { allow read, write: if isOwner(userId); match /profile/{docId} { allow read, write: if isOwner(userId); } match /builds/{buildId} { allow read, write: if isOwner(userId); } } } }
⚠️ 管理員設定 初始管理員需透過 Firebase Console 或 Admin SDK 手動在 users/{uid}/profile 中設定 role = 'ADMIN'。
安全規則使用 get() 讀取用戶角色，注意此操作計入 Firestore 讀取次數，頻繁的管理員操作不影響一般用戶的讀取配額。
06 — Indexing
複合索引策略
Firestore 單欄位自動建立索引；以下為需手動設定的複合索引（firestore.indexes.json）。
Collection欄位組合用途
pilotsclass ASC, rarity DESC依職業篩選、依稀有度排序
pilotslicense ASC, class ASC依駕照許可與職業聯合篩選
modulesslot ASC, rarity DESC依槽位篩選模組、依稀有度排序
modulesboundMechId ASC, boundPart ASC依綁定機甲查詢副模組
modulesavailable ASC, slot ASC模組圖鑑頁（僅顯示 available=true）
weaponscategory ASC, type ASC依大類與種類聯合篩選
weaponsisExclusive ASC, category ASC專武篩選
componentsslot ASC, rarity DESC依觸/應元件分類篩選
skillstags ARRAY_CONTAINS, type ASC依標籤搜尋技能
users/{uid}/buildscreatedAt DESC個人配裝依時間排序
firestore.indexes.json（部分範例）{ "indexes": [ { "collectionGroup": "pilots", "queryScope": "COLLECTION", "fields": [ { "fieldPath": "class", "order": "ASCENDING" }, { "fieldPath": "rarity", "order": "DESCENDING" } ] }, { "collectionGroup": "modules", "queryScope": "COLLECTION", "fields": [ { "fieldPath": "boundMechId", "order": "ASCENDING" }, { "fieldPath": "boundPart", "order": "ASCENDING" } ] } ] }
07 — Migration Plan
資料遷移計畫：JSON → Firestore
Phase 4（Firebase 資料庫建置）的具體執行步驟。
1
建立 Firebase 專案 + 啟用 Firestore
Firebase Console 建立專案，選擇 Cloud Firestore（Production mode），設定台灣/亞洲地區（asia-east1）。安裝 Firebase SDK：npm install firebase。
2
建立 src/types/enums.ts
將 §02 所有 Enum 定義寫入 enums.ts，同步更新 src/types/index.ts 的 interface 定義，所有字串字面量欄位替換為對應 Enum 型別。
3
建立 src/lib/firebase.ts — Firebase 初始化
初始化 Firebase App、Auth、Firestore 實例。環境變數存放 Firebase 設定（VITE_FIREBASE_API_KEY 等）於 .env.local。
4
撰寫遷移腳本：scripts/migrate-to-firestore.ts
讀取現有 JSON 檔案 → 欄位值轉換為 Enum 字串值 → 使用 Firebase Admin SDK 批量寫入（每批 500 筆，Firestore 批次限制）。腳本支援 dry-run 模式，先驗證資料格式再實際寫入。
5
部署 Firestore Security Rules + 複合索引
執行 firebase deploy --only firestore:rules,firestore:indexes。驗證：用一般帳號測試只能讀不能寫；用 Admin 帳號測試可寫入。
6
建立 Firestore 資料讀取 Hooks
src/hooks/ 下建立 usePilots、useMechs、useModules 等 React Query / SWR hooks，取代現有的 fetch('/data/pilots.json')。實作分頁、篩選、快取策略。
7
逐頁替換資料來源
依序更新：機師圖鑑 → 機甲圖鑑 → 模組圖鑑 → 武器圖鑑 → 背包圖鑑 → 配裝模擬器。每頁替換後執行 build + 功能測試，確認無回歸問題。
8
移除靜態 JSON 檔案（或保留為備份）
確認所有頁面均從 Firestore 讀取後，可將 public/data/*.json 移至 scripts/backup/ 做為歷史備份，不再部署至 GitHub Pages。
9
初始化 buffs + skills Collections
建立空的 buffs Collection（文件結構佔位），從現有 pilot 技能資料抽取共用技能寫入 skills Collection。BUFF 具體定義待遊戲資料整理後補充。
⚠️ 注意事項 • Firestore 免費方案（Spark）：50,000 讀取/天、20,000 寫入/天。網站初期規模應足夠，未來流量增長後評估升級至 Blaze 方案。
• 靜態資料建議加入前端快取（React Query staleTime / SWR revalidateOnFocus: false），避免每次頁面切換都重新讀取 Firestore。
• GitHub Pages 部署無需後端，Firebase 直接由前端 SDK 存取。請確認 Firestore 安全規則允許公開讀取後再移除 JSON 備份。

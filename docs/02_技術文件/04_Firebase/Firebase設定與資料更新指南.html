01 — Environment Variables
環境變數設定
本專案有兩份環境變數檔案，用途不同，金鑰來源也不同。兩者都不應提交至 git（已加入 .gitignore）。
⚙️ .env.local — 前端開發用（Vite + Firebase Web SDK）
📌 用途 前端 React 應用（src/lib/firebase.ts）讀取此設定初始化 Firebase Web SDK。
僅在本機開發時需要；GitHub Pages 部署透過 GitHub Actions Secrets 注入同名變數。
.env.local（複製自 .env.local.example，填入實際值） # 從 Firebase Console → Project Settings → Your apps → Web app → SDK setup 取得 VITE_FIREBASE_API_KEY=AIzaSy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx VITE_FIREBASE_AUTH_DOMAIN=your-project-id.firebaseapp.com VITE_FIREBASE_PROJECT_ID=your-project-id VITE_FIREBASE_STORAGE_BUCKET=your-project-id.firebasestorage.app VITE_FIREBASE_MESSAGING_SENDER_ID=123456789012 VITE_FIREBASE_APP_ID=1:123456789012:web:abcdef1234567890 # Cloudinary — 圖片上傳（用戶頭像） # Cloud Name：Cloudinary Console → Dashboard 左上角顯示 # Upload Preset：Cloudinary Console → Settings → Upload → Upload presets → 新增 Unsigned preset VITE_CLOUDINARY_CLOUD_NAME=your-cloud-name VITE_CLOUDINARY_UPLOAD_PRESET=your-upload-preset
1
前往 Firebase Console
開啟 console.firebase.google.com → 選擇專案（mecharashi-tools）
2
取得 Web App 設定
點選 ⚙️ Project Settings → Your apps → Web app（</> 圖示）→ SDK setup and configuration → 選 Config
3
複製並填入 .env.local
把 firebaseConfig 物件的各欄位值對應填入 .env.local.example 並另存為 .env.local
☁️ Cloudinary Upload Preset 設定（新手步驟）
📌 用途 用戶頭像上傳改用 Cloudinary，不再使用 Firebase Storage。前端以 Unsigned Upload Preset 直接將 WebP 圖片上傳至 Cloudinary CDN，取得 secure_url 後寫入 Firestore users/{uid}/profile/main.avatarUrl。
1
建立 Cloudinary 帳號
前往 cloudinary.com 免費註冊；免費方案提供 25 GB 儲存與 25 GB/月 流量，足夠初期使用。
2
取得 Cloud Name
登入後在 Dashboard 左上角可看到 Cloud name（例如 dxxxxxx），填入 VITE_CLOUDINARY_CLOUD_NAME。
3
建立 Unsigned Upload Preset
Cloudinary Console → Settings（齒輪）→ Upload → Upload presets → Add upload preset
Signing Mode 選 Unsigned → 儲存後取得 Preset name，填入 VITE_CLOUDINARY_UPLOAD_PRESET。
可選：設定 Allowed formats 為 webp,jpg,png；設定 Folder 為 avatars。
4
填入 .env.local
在 .env.local 加入 VITE_CLOUDINARY_CLOUD_NAME 與 VITE_CLOUDINARY_UPLOAD_PRESET，重啟 Vite dev server 即可生效。
⚙️ .env.migration — 遷移腳本用（Firebase Admin SDK）
📌 用途 scripts/migrate-to-firestore.mjs 使用 Firebase Admin SDK，以服務帳號身分繞過安全規則直接寫入 Firestore。
絕對不要將服務帳號金鑰 JSON 或此 env 檔提交至 git。
.env.migration（複製自 .env.migration.example，填入實際值） FIREBASE_PROJECT_ID=your-project-id # serviceAccountKey.json 的相對路徑（從 repo 根目錄計算） GOOGLE_APPLICATION_CREDENTIALS=./serviceAccountKey.json
1
產生服務帳號金鑰
Firebase Console → ⚙️ Project Settings → Service accounts → Generate new private key → 下載 JSON
2
放置金鑰檔案
將下載的 JSON 重命名為 serviceAccountKey.json 並放在 repo 根目錄（已在 .gitignore 中排除）
3
填入 .env.migration
複製 .env.migration.example 為 .env.migration，填入 FIREBASE_PROJECT_ID 和金鑰路徑
⚠️ 安全提醒 服務帳號金鑰擁有繞過所有 Firestore Security Rules 的完整寫入權限。
不要分享給不需要執行遷移腳本的人；不要上傳至任何公開位置；定期至 Firebase Console 檢查服務帳號是否有不必要的舊金鑰。
02 — Data Update Methods
資料更新流程
根據更新規模與場景選擇適合的方式。

🔄 方式 A：遷移腳本（大量更新）

推薦需 Admin 金鑰

適合：新版本遊戲資料上線、多筆資料同步更新、初次建立資料庫

  • 來源：public/data/*.json（爬蟲結果）
  • 工具：scripts/migrate-to-firestore.mjs
  • 支援 dry-run 預覽模式
  • 可指定只遷移特定 collection

✏️ 方式 B：Firebase Console 直接編輯

單筆修正無需金鑰

適合：修正個別資料錯誤、補充缺漏欄位、臨時調整單一文件

  • 在瀏覽器 Firebase Console 直接操作
  • 無需本機環境或程式碼
  • 操作量大時效率低
  • 無法批量，不留版本記錄

🕷️ 方式 C：爬蟲 → JSON → 遷移

推薦需 Admin 金鑰

適合：遊戲更新後定期同步最新機師/機甲/模組資料

  • Step 1：執行爬蟲腳本更新 JSON
  • Step 2：確認 JSON 資料正確
  • Step 3：執行遷移腳本推送至 Firestore
  • JSON 做為版本快照留存

🛠️ 方式 D：管理後台（Admin Page）

開發中需管理員帳號

適合：非技術背景的維護者直接在網站後台新增/修改單筆資料

  • 前端 UI 直接調用 firestoreApi.ts 寫入
  • 需登入且擁有 ADMIN role
  • Phase 4+ 才會實作
  • 不需要本機開發環境
🔄 方式 A：遷移腳本詳細步驟
Terminal — 遷移腳本用法 # 1. 先用 dry-run 確認資料格式無誤（不會寫入 Firestore） npm run migrate:dry # 2. 確認後執行實際遷移（全部 collections） npm run migrate # 3. 只更新特定 collection（節省寫入次數） node scripts/migrate-to-firestore.mjs --only=modules node scripts/migrate-to-firestore.mjs --only=pilots node scripts/migrate-to-firestore.mjs --only=mechs
📋 可遷移的 collections pilots、mechs、modules、weapons、backpacks、components、pilotResearch、globalResearch、enemies
資料來源一律為 public/data/{collection}.json（爬蟲或手動維護的 JSON 檔案）
🧩 modules — conditionalEffects 欄位（Array of Maps）
📌 Firestore 儲存型別 conditionalEffects 在 Firestore 中以 Array 型別儲存，陣列內每個元素為 Map（對應 ConditionalEffect 介面）。
無條件效果的模組此欄位為空陣列；副模組同樣為空陣列。
遷移腳本已自動處理預設值：modules.json 中若無 conditionalEffects 欄位，腳本會補上 [] 再寫入 Firestore。
public/data/modules.json — conditionalEffects 欄位範例 // 有條件效果的模組（每獲得增益傷害+6%，上限30%，持續2回合） { "id": "mod_1023", "name": "增益聯動Ⅱ", "dmg": 30, "conditionalEffects": [ { "trigger": "perBuffHeld", "stats": ["dmg"], "scale": 6, "max": 30, "duration": 2 } ] } // 無條件效果的模組 { "id": "mod_1001", "name": "精準射擊Ⅰ", "conditionalEffects": [] }
✏️ Firebase Console 手動新增 conditionalEffects 步驟
1
找到目標模組文件
Firestore → modules collection → 搜尋目標 Document ID（如 mod_1023）
2
新增 conditionalEffects 欄位
點 Add field → 欄位名稱輸入 conditionalEffects → 型別選 array → 確認
3
新增陣列元素（Map）
在 conditionalEffects 陣列下點 Add item → 型別選 map → 展開後逐一 Add field 填入各子欄位：
trigger（string）、stats（array of string）、scale（number）、max（number）、duration（number）等
4
同步回 modules.json
在 public/data/modules.json 找到對應模組，手動補上相同的 conditionalEffects 內容，確保下次遷移不覆蓋
⚠️ stats 欄位型別注意 Firestore Console 中 stats 必須建立為 array，陣列內每個元素型別為 string（如 "dmg"、"acc_rate"）。
若誤設為單一 string 而非 array，前端 conditionalEffect.stats.map(...) 將拋錯。
🕷️ 方式 C：爬蟲 → 遷移完整流程
Terminal — 遊戲資料更新完整流程 # Step 1：執行爬蟲更新機師資料（以機師為例） npm run scrape:v3 # 或強制重爬所有（忽略快取） npm run scrape:v3:force # Step 2：執行爬蟲更新機甲資料 npm run scrape:mechs # Step 3：確認 JSON 資料（建議用 git diff 查看異動） git diff public/data/pilots.json git diff public/data/mechs.json # Step 4：dry-run 驗證格式 npm run migrate:dry # Step 5：推送至 Firestore npm run migrate # 或只更新有變動的 collections node scripts/migrate-to-firestore.mjs --only=pilots node scripts/migrate-to-firestore.mjs --only=mechs
✏️ 方式 B：Firebase Console 直接編輯步驟
1
開啟 Firestore 資料庫
Firebase Console → 左側選單 Firestore Database → Data 標籤
2
導航至目標文件
選擇 Collection（如 pilots） → 找到目標 Document（以 Document ID 搜尋）
3
編輯欄位
點選欄位旁的鉛筆圖示直接修改值；新增欄位用 Add field；完成後點 Update
4
（可選）同步回 JSON
若 Console 修改的值未反映到 public/data/*.json，下次執行遷移腳本會被覆蓋。手動編輯 JSON 以保持一致。
03 — Quota Saving
節省 Firestore 讀取配額
Firestore Spark（免費）方案每日上限：50,000 讀取、20,000 寫入、20,000 刪除。以下是有效節省配額的策略。
操作讀取消耗說明
讀取整個 pilots collection（~50 筆）50 次每筆 document 算 1 次讀取
讀取整個 modules collection（~200 筆）200 次每次重整頁面都重新讀取的話很快消耗
讀取單一 document1 次詳情頁按需讀取效率最高
查詢不命中索引（報錯）仍計費失敗的查詢也計算讀取次數
使用 Firestore Emulator0 次（本機）完全不消耗線上配額
🔥 策略 1：使用 Firestore Emulator（最有效）
💡 原理 Firebase 提供本機 Emulator，讓開發時的所有 Firestore 讀寫都在本機進行，完全不消耗線上配額。 匯入一次線上資料快照後，後續開發都可離線進行。
Terminal — Firestore Emulator 設定與使用 # 安裝 Firebase CLI（全域，只需一次） npm install -g firebase-tools # 登入 Firebase firebase login # 初始化 Emulator（在 repo 根目錄執行，選 Firestore） firebase init emulators # 啟動 Emulator（預設 Port: 8080） firebase emulators:start --only firestore # 匯入線上資料（先匯出備份，再匯入 Emulator） firebase firestore:export ./emulator-backup firebase emulators:start --only firestore --import=./emulator-backup
src/lib/firebase.ts — 加入 Emulator 連線（開發環境自動切換） import { connectFirestoreEmulator } from 'firebase/firestore' // 在 getFirestore(app) 之後加入： if (import.meta.env.DEV && import.meta.env.VITE_USE_EMULATOR === 'true') { connectFirestoreEmulator(db, 'localhost', 8080) }
.env.local — 加入 Emulator 控制開關 # 設為 true 時連線 Emulator，false 或不設定則連線線上 Firestore VITE_USE_EMULATOR=true
⚡ 策略 2：前端快取（減少重複讀取）
💡 原理 目前的 useFirestore.ts hooks 使用 React useEffect + useState，每次元件 mount 都會重新向 Firestore 讀取。 透過以下方式可以大幅減少重複讀取。
  • ①
    Module-level 快取（最簡單）
    在 firestoreApi.ts 加入 module-level 變數快取，同一次頁面 session 內只讀取一次。
    適合不需要即時更新的靜態遊戲資料（pilots、mechs、modules 等）
  • ②
    React Context + Provider（中等複雜度）
    建立 GameDataContext 在 App 層級載入一次，所有子元件共享同一份資料。
    避免同一頁面多個元件各自觸發相同 collection 的讀取
  • ③
    React Query 或 SWR（最完整）
    引入 @tanstack/react-query，設定 staleTime（如 5 分鐘），自動處理快取、重試、背景更新。
    適合未來流量增加後的正式方案
src/lib/firestoreApi.ts — 簡單 Module-level 快取範例 // 在檔案頂部加入快取 Map const _cache = new Map<string, unknown[]>() async function fetchCollectionCached<T>(name: string): Promise<T[]> { if (_cache.has(name)) return _cache.get(name) as T[] const result = await fetchCollection<T>(name) _cache.set(name, result) return result } // 改用 fetchCollectionCached 取代 fetchCollection（靜態資料） export const getPilots = () => fetchCollectionCached<Pilot>('pilots') export const getMechs = () => fetchCollectionCached<Mech>('mechs') export const getModules = () => fetchCollectionCached<Module>('modules')
📋 策略 3：其他節省配額的好習慣
  • ✅
    開發時使用 Emulator，不要直接連線上 Firestore
    設定 VITE_USE_EMULATOR=true，讀寫完全在本機，零配額消耗
  • ✅
    遷移前一定要先用 dry-run
    npm run migrate:dry 不會寫入 Firestore，可確認資料格式再執行實際遷移
  • ✅
    只遷移有變動的 collection
    用 --only=mechs 指定 collection，不要每次都全量遷移所有資料
  • ✅
    使用 Firebase Console 的 Usage 頁面監控配額
    Firebase Console → Usage and billing → 可查看每日讀寫次數趨勢
  • ⚠️
    避免在列表頁面讀取完整 document 後再做前端篩選
    例如讀取所有 pilots 後再 filter 職業，不如使用 Firestore where 查詢或前端快取後篩選
  • ⚠️
    詳情頁盡量使用列表頁的快取資料
    從機師列表跳到機師詳情時，若列表資料已在快取，可直接使用，不需再發一次 getDoc
04 — CI/CD
GitHub Actions 部署設定
GitHub Pages 部署時需要把 Firebase 設定以 Secrets 注入，避免將金鑰硬編碼在程式碼中。
1
在 GitHub 設定 Secrets
GitHub repo → Settings → Secrets and variables → Actions → New repository secret
Firebase 相關：VITE_FIREBASE_API_KEY、VITE_FIREBASE_AUTH_DOMAIN、VITE_FIREBASE_PROJECT_ID、VITE_FIREBASE_STORAGE_BUCKET、VITE_FIREBASE_MESSAGING_SENDER_ID、VITE_FIREBASE_APP_ID
Cloudinary 相關：VITE_CLOUDINARY_CLOUD_NAME、VITE_CLOUDINARY_UPLOAD_PRESET
2
在 workflow 中使用 Secrets
確認 .github/workflows/deploy.yml 的 build 步驟有傳入 env 變數（參考以下範例）
.github/workflows/deploy.yml — build 步驟 env 設定 - name: Build run: npm run build env: VITE_FIREBASE_API_KEY: ${{ secrets.VITE_FIREBASE_API_KEY }} VITE_FIREBASE_AUTH_DOMAIN: ${{ secrets.VITE_FIREBASE_AUTH_DOMAIN }} VITE_FIREBASE_PROJECT_ID: ${{ secrets.VITE_FIREBASE_PROJECT_ID }} VITE_FIREBASE_STORAGE_BUCKET: ${{ secrets.VITE_FIREBASE_STORAGE_BUCKET }} VITE_FIREBASE_MESSAGING_SENDER_ID: ${{ secrets.VITE_FIREBASE_MESSAGING_SENDER_ID }} VITE_FIREBASE_APP_ID: ${{ secrets.VITE_FIREBASE_APP_ID }} VITE_CLOUDINARY_CLOUD_NAME: ${{ secrets.VITE_CLOUDINARY_CLOUD_NAME }} VITE_CLOUDINARY_UPLOAD_PRESET: ${{ secrets.VITE_CLOUDINARY_UPLOAD_PRESET }}
⚠️ VITE_ 前綴說明 Vite 只會將 VITE_ 開頭的環境變數暴露給前端程式碼（import.meta.env.VITE_*）。
其餘變數在 build 時不會打包進前端 bundle，可安全存放其他 CI 用途的 secrets。
05 — Troubleshooting
常見問題排查
問題可能原因解法
前端讀取資料失敗（permission-denied） Firestore Security Rules 未正確設定，或 Collection 名稱拼錯 Firebase Console → Firestore → Rules 確認 allow read: if true 已設定；檢查 collection 名稱大小寫
遷移腳本失敗：Firebase Admin 初始化失敗 .env.migration 未建立、serviceAccountKey.json 路徑錯誤 確認 .env.migration 存在並填入正確值；確認 serviceAccountKey.json 在 repo 根目錄
遷移後資料在前端沒有更新 前端快取未失效（module-level cache） 重新整理頁面（強制 F5）；或暫時清空 _cache Map
Firestore 讀取次數異常偏高 元件重複 mount、沒有快取機制、onSnapshot 即時監聽 加入 module-level 快取；避免使用 onSnapshot（改用 getDocs）；確認 React useEffect 依賴陣列正確
Firebase Console 修改了資料但遷移後被覆蓋 遷移腳本用 setDoc 覆寫整份文件 Console 修改後同步更新對應的 public/data/*.json，讓 JSON 成為唯一資料來源
build 時 Firebase 變數未注入（undefined） GitHub Actions Secrets 未設定或名稱不符 確認 Settings → Secrets 有所有 VITE_FIREBASE_* 變數；確認 workflow env 中引用了正確 secret 名稱
頭像上傳失敗（HTTP 400） Cloudinary Upload Preset 未建立、Signing Mode 設為 Signed 而非 Unsigned，或 VITE_CLOUDINARY_* 環境變數未填入 Cloudinary Console → Settings → Upload presets 確認 preset 存在且 Signing Mode 為 Unsigned；確認 .env.local 和 GitHub Secrets 有填入正確值
頭像上傳後 avatarUrl 為 undefined Cloudinary 回傳 JSON 中 secure_url 不存在（通常是 preset 設定錯誤） 在 browser DevTools Network 查看 Cloudinary API 回傳的完整 JSON；確認 preset 設定允許圖片格式（webp）

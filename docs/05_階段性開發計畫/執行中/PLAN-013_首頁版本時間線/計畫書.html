背景與動機

鋼嵐台服本質上落後陸服約半年版本，因此玩家可透過陸服已知資訊提前預測台服的改版內容與時間節點，包含新角色卡池、機甲卡池、武裝討伐、戰令、商店更新等。

玩家社群已有人工整理此類「版本預測表格」，資訊對規劃培育資源、評估角色強度極為重要。本計畫將此資訊以三個互補區塊嵌入首頁，成為工具站的核心差異化功能之一。
Page Layout
首頁整體佈局 — 三區塊設計
佈局概念：上方雙欄快覽 + 下方魚骨時間線

首頁由三個功能區塊組成：左上「版本濃縮資訊表」提供所有版本的橫向對照；右上「灰燼行動參考」列出各公司可跑機甲；下方大型「版本魚骨時間線」提供詳細互動查閱。
三個區塊資料來源均為同一個 patchVersions.ts 靜態資料，目前由管理者手動維護。

Data Categories
版本資訊分類 — 10 大類別
維護重點：哪些資訊值得追蹤？
以下為每個版本可能出現的資訊類型，不同類別有不同的出現週期，維護者需了解各類別的規律性。
# 類別 欄位名稱 出現頻率 說明
1 角色卡池 pilotBanners 每個上/下半版本各 1 個新角色 新角色首抽（上半/下半各一）；偶爾有復刻
2 機甲卡池 mechBanners 每個上/下半版本各 1 台新機甲 新機甲首抽；偶爾有復刻（白夜凜錚等）
3 角雕特遣 pilotSelection 不定期（每幾個版本一次） 舊角色二選一自選活動，規劃培育的重點
4 跨域海運 mechSelection 不定期（每幾個版本一次） 舊機甲二選一自選活動
5 角雕刮刮樂 skinGacha 幾乎每個版本 角色外觀抽取活動，含限定外觀
6 角雕輪盤 rouletteEvent 通常在版本最後一週 高性價比抽獎活動；固定週期，便於提前規劃消費
7 武裝討伐 armamentRaids 幾乎每個版本（上半/下半各可能新增） 新關卡 + 對應新武器/背包製造；攻略區未來重點
8 戰令 battlePass 每個版本必有 版本戰令，可選角色或機甲（上下半各一套）
9 商店更新 borderShop / arenaShop / crisisShop 不定期（每隔數個版本） 邊境商店、鬥技場商店；危境重構新增機師信物自選
10 灰燼行動 grayOpsUpdates 不定期新機甲加入 四家公司固定日常；機甲表（武裝工坊/創新動力/GeekX/火花塞）

欄頭（版本 × 台服日期）

  • 格式：版本號 + 台服起始日（M/D）
  • 當前版本：背景高亮（綠色系）+ ★
  • 預測版本：水藍色標示
  • 歷史版本：灰色降低顯示
  • 陸服日期不顯示（簡潔原則）

列標籤（類別）

  • 上半更新（角色+機甲）
  • 下半更新（角色+機甲）
  • 武裝生產 / 背包製作
  • 戰令（上下半各一行）
  • 危境重構 / 記憶風暴
  • 邊境商店 / 鬥技場（含台服日期）

儲存格內容規則

  • 空白表示該版本無此項目
  • 商店更新加括號標示台服日期
  • 武裝生產/背包若有多項則換行顯示
  • 戰令分兩行：角色戰令 / 機甲戰令
  • 無武裝版本明確標示「無武裝」

互動（極簡）

  • 水平滾動（版本多時），固定左側列標籤

資料結構說明

灰燼行動的機甲列表是累積型資料，每個版本的 grayOpsUpdates 記錄「本版本新加入的機甲」。
GrayOpsPanel 元件在渲染時，會掃描所有版本的 grayOpsUpdates，按公司彙整出完整的機甲列表（含版本標籤）。
不需要額外維護一份「完整機甲列表」，資料自動從版本記錄聚合。
UI Design
版本圓點展開時間線設計
核心互動概念：圓點縮圖 → 點擊展開 → 版本前瞻圖 + 雙側資訊

時間線以垂直脊線為主軸，每個大版本（如 v2.8、v3.0）對應一個圓形縮圖節點，縮圖預設顯示版本前瞻圖的壓縮小圖，版本號標示於圓點旁。

使用者點擊圓點後，節點展開為全寬面板，頂部呈現版本前瞻圖大圖，下方左右側分別顯示陸服（左）與台服（右）的更新資訊。再次點擊或點擊其他版本則收合。
頁面載入時，台服當前版本預設展開並自動捲動定位。
設計靈感來源：參考 histography.io 的圓點時間線互動模式，但大幅簡化：
僅保留「圓點縮圖 → 點擊展開」的核心效果；不需粒子動畫或其他裝飾特效。

Responsive
響應式設計方針

桌面版（≥ 768px）

  • 時間線居中，最大寬度 1200px
  • 展開後：Banner 全寬，下方左右各 50% 雙欄
  • 收合狀態：圓點 + 版本號 + 日期摘要橫排
  • 脊線為細垂直線，圓點置於其上

手機版（＜ 768px）

  • 時間線靠左，脊線為左側細線
  • 圓點縮小（直徑約 48px）
  • 展開後：Banner 全寬，下方陸服/台服垂直堆疊
  • 台服在上（玩家主要關注），陸服在下
Data Model
TypeScript 資料模型設計
維護策略：靜態 TypeScript 資料檔，手動維護陸台雙邊日期

版本資訊以靜態 TypeScript 檔案維護（src/data/patchVersions.ts），不依賴 Firebase。
每個 PatchHalf 含 cnDate（陸服精確日期）與 twDate（台服日期，確認或預測）兩個獨立欄位。
陸服更新後由維護者填入精確日期；台服預測日期填入後標記 twIsPredicted: true，待台服確認後移除此旗標。
TypeScript 型別保護確保欄位格式正確；維護者只需在陣列末尾 append 新版本物件。

模組化設計原則
  • VersionTimeline 以 useState<string|null> 管理目前展開版本號，點擊同版本則收合（toggle）
  • VersionDot 為純 UI，不持有狀態，透過 onToggle prop 回呼父層
  • VersionExpandedPanel 接受整個 PatchVersion，自行渲染 upper + lower 雙半
  • VersionSidePanel 接受 version: PatchVersion + side: 'cn'|'tw'，純 props 驅動
  • 資料 (patchVersions.ts) 與呈現 (components) 完全分離；日後換 Firebase 只需替換資料來源

Maintenance
維護者操作指南
維護流程：陸服更新時
  1. 打開 src/data/patchVersions.ts
  2. 在對應版本的 upper 或 lower 中填入正確的 cnDate（精確日期）
  3. 更新 pilots、mechs、specialEvents 等內容欄位
  4. 若同時更新台服預測：修改 twDate，並確認 twIsPredicted: true
  5. 儲存後執行 npm run build 確認無型別錯誤，推送 GitHub
維護流程：台服版本上線時
  1. 將上線版本的 twDate 改為實際日期（移除「約」字）
  2. 移除 twIsPredicted: true（或改為 false），UI 自動改為確認樣式
  3. 將舊版本的 isTwCurrent: true 移除，改設到新的當前版本
  4. 儲存後執行 npm run build，推送 GitHub
常見情境
  • 新版本前瞻圖：圖片放至 public/images/banners/vX.X.jpg，資料填 bannerImage: '/images/banners/vX.X.jpg'；無圖時圓點顯示佔位色塊，不影響功能
  • 台服日期未確認：twDate: '約 2026/07' + twIsPredicted: true；右側以水藍色顯示
  • 台服日期已確認：填入精確 twDate: '2026/07/16'，移除 twIsPredicted
  • 武裝討伐有新增但不清楚詳情：armamentRaids: [{ name: '（待確認）' }]
  • 角雕輪盤確認：rouletteEvent: true，UI 自動顯示「⭐ 角雕輪盤（末週）」提示
  • 陸服有多個子更新：寫入 specialEvents: ['2026/05/21 限時簽到活動', '2026/05/28 角雕輪盤']
  • 危境重構有新信物：crisisShop: ['機師A信物', '機師B信物']
  • 灰燼行動新機甲：grayOpsUpdates: [{ company: '武裝工坊', newMechs: ['霸王'] }]

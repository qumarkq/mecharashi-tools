◆ PLAN-004

技能資料抽離pilotSkills · neuralDrives · PilotTalent 保留

將 PilotSkill 與職業共用 NeuralDrive 從 Pilot 文件中抽離為獨立 Collection，解決共用被動重複維護問題，同時釐清技能/天賦/神經驅動的管理邊界

Problem
問題背景
技能資料目前全數嵌入於 Pilot 文件，隨著遊戲機師增加，已暴露三個結構性維護問題。
問題一：共用被動技能需要在多份 Pilot 文件中各自維護一份副本

現有 Pilot.skills: PilotSkill[] 將所有技能直接嵌入機師文件。 當多名機師共用同一個被動技能（相同名稱、相同效果）時，該技能的 description、effects[]、buffIds[] 等欄位在每份 Pilot 文件中各有一份副本。

若日後遊戲更新導致某個共用被動的數值調整，需逐一找出並更新所有持有該技能的機師文件， 既費時又容易漏改，且無法保證一致性。隨著機師數量增加，此問題的維護成本將持續上升。
問題二：神經驅動職業共用 / 機師專屬混存，現有結構無法區分

Pilot.neuralDrive: NeuralDrive[] 嵌入所有神經驅動區塊。 部分神經驅動區塊（如某些 α / β 分區）為同職業所有機師共用（相同激活閾值、相同效果）， 部分則為特定機師專屬（γ 分區常見此情況）。

現有 NeuralDrive interface 沒有任何欄位說明此區塊的歸屬（職業共用 vs 機師專屬）， 無法在 Admin 工具中做結構化管理。職業共用的部分同樣面臨問題一的重複維護問題—— 同職業多位機師各存一份，更新時必須全部手動同步。
問題三：effects[] 結構化數值填寫進度無法集中追蹤

計算器依賴每個技能的 effects: SkillEffect[] 做數值加總。 目前技能分散在各 Pilot 文件中，無法在單一位置概覽「哪些技能已填完 effects[]」、 「哪些仍為空陣列待補」。若要做 Admin 工具批次補填，必須逐份掃描 Pilot 文件， 極為不便。
Scope
範圍界定
三種技能相關資料的處理方式不同，根據各自特性決定是否抽離。
資料類型現有位置處置原因
PilotSkill Pilot.skills[]（嵌入） ✓ 抽離 → pilotSkills collection 存在跨機師共用情況；需單一管理入口追蹤 effects[] 補填進度
NeuralDrive Pilot.neuralDrive[]（嵌入） ✓ 抽離 → neuralDrives collection 存在職業共用 / 機師專屬之分；需 scope 欄位結構化區分
PilotTalent Pilot.talents[]（嵌入） ✗ 保留嵌入 天賦為機師個別屬性；enhancedEffects 依賴專武觸發，本質上與機師文件強耦合，抽離無實際效益
天賦不抽離的補充說明
天賦名稱與效果在每個機師中均唯一，不存在跨機師共用的情況。 武器端透過 WeaponSkill.enhancesTalentName 來對應要強化的天賦， 而強化後的效果 (enhancedEffects) 儲存於 Pilot 文件中，是機師個別配置的一部份。 此設計已在 PLAN-001 中確立，本次不調整。

推薦方案 A 的核心理由
方案 A 的前端改動看似大，但 pilotSkills 和 neuralDrives 屬於靜態參考資料（與遊戲版本同步，不常變動）， 可在 App 啟動時一次性預載並快取到 React Context 或 Zustand store， 之後任何頁面查詢技能時都是 O(1) 的 Map lookup，效能不劣於嵌入方案。 方案 B 的「同步」機制反而引入額外的操作風險（忘記同步、只改了 master 沒推送）， 是一種把問題藏起來而非真正解決的做法。
Design
設計決策
採用方案 A（全抽離 + 全域快取），以下說明各項具體決策。
決策一：PilotSkill 全數抽離至 pilotSkills Collection

將 Pilot 文件中的 skills[] 陣列拆解為獨立文件，每份文件對應一個技能（含被動、主動、指令、武器技能）。 技能 ID 採 slug 格式：{type}_{技能名稱}（如 passive_突擊強化）。

去重策略：名稱相同且 description 一致者視為同一技能，共用一份文件。 名稱相同但 description 不同（同名不同效）者，視為不同文件，以 _{pilotId} 後綴區分。

Pilot.skills 欄位由 PilotSkill[] 改為 string[]（pilotSkills 的文件 ID 陣列，順序保留）。
決策二：NeuralDrive 全數抽離至 neuralDrives Collection，新增 scope 欄位

每個神經驅動區塊（如「進攻者α」、「技術者γ」）獨立為一份文件。 新增 scope: 'pilot' | 'class' 欄位區分歸屬：
· scope = 'class'：職業共用，搭配 pilotClass: string 記錄適用職業
· scope = 'pilot'：機師專屬，搭配 pilotId: string 記錄所屬機師

Pilot.neuralDrive 欄位由 NeuralDrive[] 改為 string[]（neuralDrives 的文件 ID 陣列）。
決策三：PilotTalent 保留嵌入，不納入本次計畫

天賦是機師個別屬性，不存在跨機師共用的情況。 enhancedEffects 為機師持有專武時的強化效果，與機師文件強耦合。 保持現狀可避免引入不必要的關聯複雜度。

若未來有「複製天賦到新機師」的需求，可再評估，但目前不在範圍內。
決策四：前端採用全域預載 + Map 快取，不做 per-pilot lazy fetch

pilotSkills 與 neuralDrives 屬於靜態參考資料，整個 Collection 體積小（預估數百條記錄）。 在 App 啟動時一次性 fetch 並建立 Map<id, object>， 之後所有頁面（PilotDetailPage、SimulatorPage 等）直接從 Map 查詢， 無需發送額外 Firestore 請求，效能等同嵌入方案。

快取層位置：React Context（SkillCacheContext）或 Zustand global store， 具體實作由開發時決定，本計畫不限定。
決策五：遷移採兩階段策略，先建新 Collection 再更新 Pilot docs

Stage 1：遷移腳本讀取所有 Pilot docs，提取 skills / neuralDrive，去重後寫入 pilotSkills / neuralDrives Collection，不修改既有 Pilot docs（保留舊格式，可 dry-run 驗證）。
Stage 2：確認新 Collection 資料正確後，執行第二腳本將 Pilot.skills / neuralDrive 欄位改為 ID 陣列。

兩個階段之間可做完整的資料驗證，降低遷移失敗風險。
Schema
Schema 規格
新增與修改的 TypeScript interface 完整定義。
1. 新增：PilotSkillDoc interface（pilotSkills collection）
// pilotSkills Collection：每份文件為一個可共用的技能 export interface PilotSkillDoc { id: string // slug，如 "passive_突擊強化" name: string // 技能顯示名稱 /** SkillType：被動技能 / 主動技能 / 指令技能 / 必殺技能 / 武器技能 */ type: string /** 仿生電腦單元類型："0"=核心單元 "6"=職業單元 */ unitType?: string ap?: string // 主動 / 指令 / 必殺技能才有 cd?: string // 指令技能才有（冷卻回合數） weapon?: WeaponRequirement | string // 武器技能才有 description: string icon: string iconLocal: string effects: SkillEffect[] // 空陣列 = 待填入 buffIds: string[] }
PilotSkillDoc 與現有 PilotSkill interface 的差異
兩者欄位幾乎相同，差別在於 PilotSkillDoc 是 Firestore 文件（有頂層 id）， 而 PilotSkill 是嵌入陣列元素。 遷移完成後，PilotSkill interface 可以直接改為 PilotSkillDoc 的別名或刪除。
2. 新增：NeuralDriveDoc interface（neuralDrives collection）
// neuralDrives Collection：每份文件為一個神經驅動區塊 export interface NeuralDriveDoc { id: string // slug，如 "class_突擊手_alpha" 或 "pilot_赤瞳_gamma" name: string // 區塊顯示名稱（如「進攻者α」） icon: string slots: string[] // 插槽類型說明列表 levels: NeuralDriveLevel[] // ─── 新增：歸屬範圍 ────────────────────────────────────────────── /** 'class' = 職業共用；'pilot' = 機師專屬 */ scope: 'class' | 'pilot' /** scope='class' 時：適用職業（PilotClass enum 值，如 "突擊手"） */ pilotClass?: string /** scope='pilot' 時：所屬機師 ID */ pilotId?: string }
3. 修改：Pilot interface（index.ts）
export interface Pilot { id: string name: string // ... 其他欄位不變 ... talents: PilotTalent[] // 不變（保留嵌入，機師個別天賦） // ─── 改變 ────────────────────────────────────────────────────── /** 原 PilotSkill[]，改為 pilotSkills Collection 的文件 ID 陣列（順序保留） */ skills: string[] /** 原 NeuralDrive[]，改為 neuralDrives Collection 的文件 ID 陣列（順序保留） */ neuralDrive: string[] }
4. 前端快取結構建議（App 層）
// App 啟動時預載，建立快速 lookup Map const skillMap = new Map<string, PilotSkillDoc>() // id → skill const neuralDriveMap = new Map<string, NeuralDriveDoc>() // id → drive // PilotDetailPage 使用方式 const pilotSkills = pilot.skills.map(id => skillMap.get(id)).filter(Boolean) const pilotDrives = pilot.neuralDrive.map(id => neuralDriveMap.get(id)).filter(Boolean)

Risk
風險分析
⚠ HIGH — 前端讀取路徑全面修改
Pilot.skills 與 neuralDrive 欄位型別由物件陣列改為 ID 字串陣列， TypeScript 編譯器會在全域標出所有受影響的使用位置（PilotDetailPage、SimulatorPage 等）， 需逐一修改。遺漏修改會導致運行時 undefined 錯誤。
緩解：TypeScript 型別改動是自我揭露的——編譯失敗代表修改不完整。 先跑 tsc --noEmit 確認所有錯誤，再逐一修正。 全域快取層統一封裝 lookup 邏輯，前端不直接存取 Map，降低遺漏機率。
⚠ HIGH — 遷移期間資料狀態不一致
Stage 1（新 Collection 寫入）完成後、Stage 2（Pilot docs 更新）執行前， 新舊格式並存。若此期間前端已改成讀 ID 陣列， 但 Pilot docs 仍是嵌入格式，會導致頁面空白。
緩解：前端改動（Interface / fetch 路徑）應在 Stage 2 完成後才部署， 不能在遷移中途上線。建議整個 PLAN-004 在非主要開發週期中的短維護窗口執行。
⚡ MEDIUM — 去重邏輯判斷失誤
共用被動的識別依賴「名稱相同 + description 相同」。 若遊戲中存在名稱相同但效果微妙不同的技能版本（如機師升星後技能升級）， 去重邏輯可能將其錯誤合併，導致部分機師技能資料遺失。
緩解：遷移腳本 dry-run 後人工審查輸出的去重報告； 有疑慮的同名技能先各自保留，標記為待確認。
⚡ MEDIUM — 全域快取首次載入延遲
App 啟動時需一次性 fetch pilotSkills / neuralDrives 兩個 Collection， 若 Collection 體積成長到數千條記錄， 首次載入時間可能明顯增加。
緩解：目前遊戲機師數量有限，預估 pilotSkills < 500 筆、 neuralDrives < 200 筆，單次 fetch 在合理範圍內。 若未來規模成長，可評估改用 Firestore 分頁或 CDN 靜態 JSON。
✓ LOW — TypeScript 型別改動
Pilot interface 的 skills / neuralDrive 欄位型別改變， 所有引用位置會在編譯期產生型別錯誤，確保不會有遺漏。
這是有利風險——讓靜態分析幫你找出所有需要修改的地方， 相比嵌入方案的無型別保護更安全。
✓ LOW — 神經驅動 scope 填值錯誤
遷移時需要人工判斷每個 NeuralDrive 區塊是 class 還是 pilot scope。 若判斷錯誤，class-shared 的區塊可能被當作 pilot-specific， 導致其他機師缺少該驅動。
緩解：可透過比對同職業多名機師的 neuralDrive 陣列， 找出相同 name + levels 的區塊，自動標記為 class scope。 人工確認例外情況。

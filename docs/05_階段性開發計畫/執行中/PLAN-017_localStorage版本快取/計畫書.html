◆ PLAN-017

localStorage 版本快取跨 Session · VersionCheck · ZeroReads

在 PLAN-016 的基礎上，讓回訪用戶的遊戲資料完全從 localStorage 讀取，每次 session 僅消耗 1 次 Firestore 讀取（版本檢查），估計降低回訪流量 91%

Problem
PLAN-016 未解決的問題
PLAN-016 的成果： 建立 GameDataContext，讓同一個 session 內頁面切換不再重複讀取 Firestore。 每次 session 從 20–30 次讀取降至 11 次（一次性讀取 8 個 collection）。
剩餘問題：每次新 session 仍需讀取 11 次
用戶每次開新分頁、重整頁面、或關閉後重開，React 重新掛載，GameDataContext 重新從 Firestore 讀取全部 collection。 遊戲資料（pilots、mechs、weapons 等）屬準靜態資料，版本更新週期以週計算， 絕大多數 session 讀到的資料和上次完全相同，這 11 次讀取毫無必要。
規模試算：回訪用戶佔多數時讀取仍然爆量
假設每日有 100 位活躍用戶，每人平均 3 次 session（開新 tab / 重整）：
300 session × 11 reads = 3,300 次/天，佔免費配額 6.6%。
若用戶增長至 500 人或重度用戶每日多次重整，壓力仍然顯著。
PLAN-017 實施後：300 session × 1 read = 300 次/天，降幅 91%。
Design
設計決策
決策一：Firestore 新增 meta/gameData 版本文件
在 Firestore 新增 meta collection，其中 gameData 文件只存儲： { version: string, updatedAt: Timestamp }。
版本號格式採用語義化版本（如 "1.0.5"），每次有任何 game collection 資料變動時， admin 手動或腳本 bump 此版本號。
此文件極輕量，讀取成本幾乎可忽略（1 document read = 1 單位）。
決策二：GameDataContext 啟動時先讀版本，再決定是否走 localStorage
啟動流程：
  1. 讀取 meta/gameData（1 次 Firestore read）
  2. 比對 localStorage.getItem('mecharashi_game_version')
  3. 版本相同 → 從 localStorage.getItem('mecharashi_game_data') 解析資料，跳過 11 次讀取
  4. 版本不同 / 無快取 → 讀取全部 collection，完成後寫入 localStorage

若 meta/gameData 文件不存在（尚未設置），退化為 PLAN-016 行為（直接讀取全部）。
決策三：localStorage 儲存格式採 JSON 序列化
以 JSON.stringify 將 GameDataState 的資料部分（pilots、mechs 等 8 個陣列） 存入單一 key mecharashi_game_data。版本號獨立存於 mecharashi_game_version。
反序列化時重建 TypeScript 物件陣列，型別相容性由現有 Firestore 資料格式保證。
localStorage 儲存上限一般為 5–10 MB。遊戲資料預估 200–500 KB，安全範圍內。
決策四：localStorage 解析失敗時靜默降級
若 JSON.parse 失敗或資料結構異常，清除 localStorage key 並回退至正常 Firestore 讀取流程。 不拋出 UI 錯誤，用戶感知不到降級，只是多付出一次正常的讀取成本。
決策五：Admin 版本 bump 機制
Phase B 提供兩種 bump 方式：
1. Admin Panel 按鈕：在現有 admin 頁面新增「更新遊戲資料版本」按鈕， 點擊後寫入新的 version（timestamp-based，如 "2026.05.27.1"）。
2. 資料腳本自動 bump：在資料匯入腳本末尾自動執行 version bump， 確保每次資料更新後快取自動失效。
兩種方式不互斥，視實際管理流程選擇。
邊界情況：版本 bump 後用戶仍有舊快取
版本 bump 後，用戶下次開啟頁面時會讀取新版本號並發現 mismatch，自動重新讀取全部 collection 並更新 localStorage。 不需要額外的強制清除機制，版本號比對自然處理。
Analysis
優化前後讀取比較
場景 PLAN-016 後（現況） PLAN-017 後 降幅
首次訪問（無快取） 11 reads 12 reads（1 version + 11 data） +1（可接受）
回訪（版本未更新） 11 reads 1 read（version check only） -91%
版本更新後首次訪問 11 reads 12 reads（cache miss，重拉） +1（可接受）
meta/gameData 不存在（降級） 11 reads 11 reads（退化行為） 無變化
整體效果估算（以 100 日活用戶、每人 3 session/天 計算）：
目前（PLAN-016 後）：300 session/天 × 11 reads = 3,300 reads/天
PLAN-017 後（假設 90% 為回訪）：270 × 1 + 30 × 12 = 630 reads/天
總降幅 81%，免費配額（5 萬/天）可支撐 ~24,000 session/天（約 8,000 活躍用戶/天）。

// GameDataContext 啟動流程（PLAN-017 後） async function loadGameData() { // Step 1: 讀版本文件（1 Firestore read，極便宜） const metaDoc = await getDoc(doc(db, 'meta', 'gameData')) const remoteVersion = metaDoc.exists() ? metaDoc.data().version : null // Step 2: 比對本地快取版本 const localVersion = localStorage.getItem('mecharashi_game_version') if (remoteVersion && localVersion === remoteVersion) { // Step 3a: 版本相同 → 從 localStorage 還原（0 Firestore reads） try { const cached = localStorage.getItem('mecharashi_game_data') if (cached) return JSON.parse(cached) } catch { // 解析失敗 → 靜默降級，繼續正常讀取 localStorage.removeItem('mecharashi_game_data') localStorage.removeItem('mecharashi_game_version') } } // Step 3b: 版本不同 / 無快取 → 讀取全部 collection const data = await Promise.all([...]) // 11 reads // Step 4: 寫入 localStorage if (remoteVersion) { localStorage.setItem('mecharashi_game_version', remoteVersion) localStorage.setItem('mecharashi_game_data', JSON.stringify(data)) } return data }
// Firestore meta/gameData 文件結構 interface GameDataMeta { version: string // e.g. "2026.05.27.1"（日期 + 當日流水號） updatedAt: Timestamp // 最後更新時間 updatedBy: string // admin uid，方便追蹤 } // localStorage keys MECHARASHI_GAME_VERSION = 'mecharashi_game_version' // 存版本號字串 MECHARASHI_GAME_DATA = 'mecharashi_game_data' // 存 JSON.stringify 的資料物件
Scope
實作範圍與不在範圍內
在範圍內：
  • GameDataContext 加入 localStorage 快取邏輯（讀取 + 寫入 + 降級）
  • Firestore meta/gameData 文件初始設置（admin 一次性作業）
  • Admin Panel 新增「Bump 遊戲資料版本」按鈕
  • 資料匯入腳本末尾自動執行 version bump（可選）
不在範圍內：
  • 用戶資料（builds、profile）的 localStorage 快取——用戶資料是動態的，不適合此策略
  • Service Worker / Cache API——Firestore WebChannel 不可被 SW 攔截，成本高效益低
  • patchVersions 的快取——此 collection 已在 PLAN-016 改為 getDocs，影響小
  • IndexedDB Firestore persistence——難以控制失效，不如自管 localStorage 透明

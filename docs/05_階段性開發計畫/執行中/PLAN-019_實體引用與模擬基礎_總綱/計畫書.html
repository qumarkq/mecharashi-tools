實體引用與模擬基礎EntityRef · Buff Graph · BOSS / Stage

定位
Master Plan（大綱）
一句話：一個實體的描述文字常引用其他實體（如天賦「悖想先驅」內含 [虛粒子形態]、[激發能]、[形態增益]、[粒子爆發]、[海莉絲]）， 但現行 description 都是純字串，系統看不見這些引用。本總綱把引用變成型別化、可解析、可模擬的資料， 並以此為大綱拆出子計畫。
Problem
問題背景 — PLAN-001/003/004/005 的同一個問題
已解決的一半（數值）：PLAN-001 已用 SkillEffect[]（stat/value/scope/condition）、 GameBuff（buffs 集合）、buffIds[] 表達技能的「可計算數值」。PLAN-003/005 補上武器側與專武強化。
未解決的一半（引用關聯）：description 內 [xxx] 這類跨實體連結被困在字串裡。 系統無法把它變成可點擊、無法驗證引用是否存在、無法在模擬器中把它當成語義邊處理。 PLAN-010 Phase B 其實發明了 EntityReference 解法（entityType+entityId、批次預載、點擊→Drawer）， 但只用在攻略文章。本總綱把它一般化到所有 description。
更深一層（使用者澄清）：引用不只是「顯示連結」。天賦/技能延伸的特殊效果， 可能是既有屬性（增傷/加子彈/加暴傷）、可能是特殊 BUFF、也可能是機師專屬的型態切換。 未來傷害模擬最合理的做法，是讓使用者設定的模組/機甲/機師去觸發特定 BUFF， 才不會發散出「不可能存在」的組合。因此抽象化比單純顯示更重要。
Architecture
核心架構：四層抽象
Layer 1 — 統一型別化引用 EntityRef（輕量、表單友善、可升級）
不把 description 改成節點樹（採「先表單」路線）。description: string 維持遊戲原文唯一真相 （[xxx] 標記原樣保留，符合 PLAN-005「原文不動」），外掛一份解析側錄 descriptionRefs。 顯示層寫一個 <RefText> tokenizes [xxx] → 查表命中渲染可點擊（點擊開 Drawer，複用 PLAN-010）； 未命中原樣顯示，優雅降級。日後換 TipTap 時，節點 attrs 即是 EntityRef，資料零浪費。
// src/types/index.ts —— 已落地 export type RefType = | 'buff' | 'skill' | 'pilot' | 'mech' | 'weapon' | 'module' | 'backpack' | 'component' | 'stat' | 'term' export interface EntityRef { refType: RefType refId: string // buffs/{id}、pilots/{id}、glossaryTerms/{id}，或 stat key 'dmg' label?: string // 顯示文字；預設取目標 name } // description 內 [xxx] → 對應實體。key = 括號內文字，如 "虛粒子形態" export type DescriptionRefs = Record<string, EntityRef>
為何「混合」而非單一詞條庫或純型別化： 增傷/加子彈/加暴傷 = refType:'stat'（既有屬性 key）；特殊增益/狀態 = 'buff'； [海莉絲] = 'pilot'/'mech'；[固定傷害] 這種無數值歸宿的機制關鍵字 = 'term'（glossaryTerms）。 一個 resolver 依 refType 分派；glossaryTerms 只收「無專屬集合可放」的關鍵字。
Layer 2 — BUFF / 形態 的模擬可達性圖（抽象優先）
形態（型態切換）= GameBuff with buffType:'state' + mutexGroup:'<pilotId>_forms'； 形態各自帶 effects[]（形態增益），同群一次只能存在一個（虛粒子形態 ⟷ 實粒子形態）。

觸發來源圖 = 沿用既有 buffIds[] 作為唯一真相的「賦予邊」：凡會產生 BUFF 的 技能/天賦/神經驅動/模組/武器，其 buffIds[] 即宣告「我能賦予這些 buff」。 模擬器建立反向索引，可用 buff = 當前配裝所有實體 buffIds[] 的聯集，再依 mutexGroup 收斂為同群擇一。
// GameBuff 擴充（已落地） export interface GameBuff { // ... 既有：id/name/description/icon/buffType/maxStack/duration/effects ... descriptionRefs?: DescriptionRefs mutexGroup?: string // 形態互斥群組；同 group 一次一個 } // Module 擴充：補上 buffIds（武器/技能/天賦/神經驅動已有） interface Module { /* ... */ buffIds?: string[] }
模擬器可達性流程（杜絕不可能組合）： 使用者配裝 機師 + 機甲 + 模組×N + 武器 + 背包 │ 收集各實體的 buffIds[] ▼ 可用 buff 池 = ⋃ buffIds[] ← 只有「來源在配裝中」的 buff 才會出現 │ 依 mutexGroup 分群 ▼ 形態/狀態 同群擇一 → 使用者勾選 active → SkillEffect[] 加總 → 面板輸出
Layer 3 — BOSS / 關卡資料模型（吸收 bossDrops.ts）
現行 src/data/bossDrops.ts 是靜態的 元件→關卡→掉落索引。一般化為 Firestore 集合 bosses / stages，並複用既有結構（ArmorType、MechPart 數值面、SkillEffect、EntityRef）。 BOSS 技能描述同樣吃 Layer 1 引用層 → 零額外機制即可點擊查詞條。遷移保留相容檢視函式給現有 BossDropTooltip。
// bosses/{id}、stages/{id}（已落地型別） interface Boss { id; name; armorType?; parts?; skills: BossSkill[]; descriptionRefs?; weakness?[] } interface Stage { id; name; category; bossIds: string[]; mechanics?; mechanicsRefs?; drops?: StageDropEntry[]; guideIds?: string[] } // guideIds → PLAN-010 攻略
Layer 4 — 維護模式與查詢量控制（沿用既有架構）

非技術維護（他人加入）

  • 沿用 PLAN-014 表單式 Admin + UserProfile.role（已有 USER/ADMIN/OWNER）
  • 新集合各給表單頁，description 旁放「插入引用」詞條挑選器（Layer 1）
  • 先表單、日後 TipTap（使用者 Q3 決策）
  • 稽核：複用 managedBy 慣例 + updatedBy/updatedAt；沿用補填進度儀表

查詢量控制（明確需求）

  • 參考實體皆為靜態資料 → PLAN-016 GameDataContext 啟動預載成 Map，頁面 O(1) 查詢
  • PLAN-017 meta/gameData 版本 gate + localStorage：回訪每 session 僅 1 次讀取
  • Admin 一鍵 Bump 版本使全站快取失效
  • 規模再大可建置期匯出 CDN 靜態 JSON，讀取成本趨近 0

Verification
驗證方式
  1. 型別：npm run build（tsc -b && vite build）通過；新欄位皆選填不破壞既有頁面。
  2. 引用層：天賦填 descriptionRefs（"虛粒子形態"→{refType:'buff',refId:'buff_虛粒子形態'}），前台該詞可點擊開 Drawer；未填的詞原樣顯示不報錯。
  3. 模擬可達性：粒子爆發 技能 buffIds:['buff_虛粒子形態'] + 該 buff mutexGroup；選該機師才出現此形態，同群互斥擇一。
  4. BOSS/關卡：跑遷移腳本後 BossDropTooltip 經相容檢視仍正常；新 stages 頁能列關卡→BOSS→掉落。
  5. 查詢量：參考集合走預載 Map + localStorage 版本快取，重整頁面不重抓未變更資料。
  6. 維護者流程：非 OWNER 的 ADMIN 帳號用表單 + 詞條挑選器新增 glossaryTerm 並在前台引用，全程不碰程式碼。

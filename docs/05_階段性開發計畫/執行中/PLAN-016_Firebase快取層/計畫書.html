Firebase 快取層GameDataContext · QueryOptimize · ReadReduction

Problem
問題背景
事件：2026-05-26 Firestore 讀取達 16 萬次，超出免費配額上限 (5 萬/天) 的 3.2 倍。
Firebase 主控台顯示寫入僅 42 次（正常），讀取爆炸為唯一異常。
根本問題一：無跨組件快取層
src/hooks/useFirestore.ts 內每個 hook（usePilots、useMechs 等）都維護獨立 state。 每次切換頁面（首頁 → 機師頁 → 模擬器）React 重新掛載組件，各自觸發新的 Firestore 讀取， 不存在任何跨頁面的資料共享或 deduplication。
根本問題二：useAllGameData() 每次進入模擬器讀 8 個 collection
src/hooks/useFirestore.ts:326–359 — 模擬器頁面觸發同時讀取： pilots、mechs、weapons、backpacks、modules、components、pilotResearch、globalResearch。 這是 8 次完整 collection 讀取，每次進入模擬器就燒掉 8+ 個配額單位。
根本問題三：首頁 VersionQuickTable 讀 4 個 collection
src/components/home/VersionQuickTable.tsx:200–203 — 僅為了建立圖片 URL 對照表， 就讀取完整的 pilots、mechs、weapons、backpacks 四個 collection。
查詢問題一：專武查詢讀全部武器再 client filter
src/hooks/useFirestore.ts:218–254 — usePilotExclusiveWeapon/s 讀取全部 100+ 武器， 再以 .find(w => w.exclusiveFor === pilotId) client-side 過濾， 應改用 Firestore where('exclusiveFor', '==', pilotId)，只讀 1–2 筆。
查詢問題二：useMechWithModules 讀全部模組再 filter
src/hooks/useFirestore.ts:132–165 — getModules() 讀取所有模組再 client-side 篩選， 明明已有 getModulesByMech(mechId) 可用 WHERE 查詢直接取得。
查詢問題三：usePatchVersions 對靜態資料使用 real-time listener
src/hooks/usePatchVersions.ts:18–39 — onSnapshot 對幾乎不更新的 patchVersions collection 建立持續監聽，應改為 getDocs 一次性讀取。
Analysis
讀取量化分析
問題 觸發時機 讀取數/次 影響
無快取層（每次頁面切換重讀） 每次 React 組件掛載 3–8 次/頁面 CRITICAL
useAllGameData() 模擬器進入 每次進模擬器 8 次/次 CRITICAL
VersionQuickTable 首頁讀 4 個 collection 每次開首頁 4 次/次 HIGH
usePilotExclusiveWeapon/s 讀全部武器 每次開機師詳情頁 100+ 文件/次 HIGH
useMechWithModules 讀全部模組 每次開機甲詳情頁 100+ 文件/次 HIGH
usePatchVersions onSnapshot 組件存在時持續監聽 持續計費 MEDIUM
getUserBuilds 每次切換 tab 重讀 每次切換到 builds tab 1+ 次/切換 LOW
估計效果： 現況每次 session 約 20–30 次 Firestore 讀取；實作 GameDataContext 後降至 5–8 次。 整體降幅 70–80%，免費配額應可支撐 8,000–10,000 次頁面操作/天。
Design
設計決策
決策一：建立 GameDataContext，App 啟動時一次讀取所有 game data
新增 src/contexts/GameDataContext.tsx，在 Provider 掛載時並行讀取 11 個 game data collection， 結果存入 Context state。所有組件改從 useGameData() 取資料，不再各自呼叫 Firestore hooks。 整個 session 只有一次初始讀取，頁面切換完全走 in-memory cache。

遊戲資料（pilots、mechs、weapons 等）屬「準靜態」資料，每次版本更新才會變動， session 內不需要 real-time sync，Context 快取完全合適。
決策二：useFirestore.ts hooks 改為讀取 GameDataContext
原有 usePilots()、useMechs() 等 hooks 保留 API 不變（降低修改面）， 內部改為讀取 useGameData().pilots 等，不再觸發 Firestore 呼叫。 現有呼叫這些 hooks 的組件不需修改。
決策三：usePilotExclusiveWeapon/s 改用 Firestore WHERE query
查詢改為 query(weaponsCollection, where('isExclusive', '==', true), where('exclusiveFor', '==', pilotId))。 因為 GameDataContext 已有全部武器快取，這兩個 hooks 也可直接從 Context filter， 不需要新的 Firestore 讀取。優先採用 Context filter 方案，更簡單且無額外網路請求。
決策四：useMechWithModules 改用 Context 中的 modules 過濾
GameDataContext 內已有完整 modules 陣列，useMechWithModules 改為從 Context filter modules.filter(m => m.boundMechId === mechId)， 不再呼叫 getModules()（讀整個 collection）或 getModulesByMech()（仍需讀 Firestore）。
決策五：usePatchVersions 改為 getDocs 一次性讀取
移除 onSnapshot，改用 getDocs。patchVersions 資料每次版本更新才會異動， 用戶在 session 內不需要看到即時更新，一次讀取即可。 若未來需要即時更新可在 Context 層統一加入 polling 策略。
決策六：getUserBuilds 加入 hasLoaded 旗標，避免 tab 切換重讀
在 ProfilePage 的 builds state 旁加 buildsLoaded: boolean， 已讀取過一次後切換 tab 不再重複 fetch，直到用戶登出或強制刷新。
Architecture
快取層架構
// 現況（有問題）：每個 hook 獨立讀 Firestore PilotsPage → usePilots() → getDocs(pilots) // 1 read SimulatorPage → useAllGameData() → getDocs(x8) // 8 reads HomePage → usePilots() → getDocs(pilots) // 重複 1 read // 目標（GameDataContext）：App 啟動時一次讀取，全程 in-memory App └─ GameDataProvider → getDocs(all 11 collections) // 1 次，僅啟動時 ├─ PilotsPage → useGameData().pilots // 0 reads ├─ SimulatorPage → useGameData().all // 0 reads └─ HomePage → useGameData().pilots // 0 reads
// GameDataContext 型別草稿 interface GameDataState { pilots: Pilot[]; mechs: Mech[]; weapons: Weapon[]; backpacks: Backpack[]; modules: Module[]; components: Component[]; skills: Skill[]; buffs: Buff[]; pilotResearch: PilotResearch[]; globalResearch: GlobalResearch[]; enemies: Enemy[]; loading: boolean; error: string | null; reload: () => void; // 強制重讀（admin 更新資料後使用） }

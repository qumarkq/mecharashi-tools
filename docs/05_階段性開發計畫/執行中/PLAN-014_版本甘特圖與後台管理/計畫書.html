背景與動機

PLAN-013 已完成版本時間線的核心展示（圓點展開 + 陸台雙欄資訊），但目前所有活動（復刻卡池、限時活動、刮刮樂等）只有「屬於上半或下半」的模糊時間資訊，無法得知活動的實際開始日與持續時間。

此外，原始資料以 TypeScript 靜態檔案維護，需要有程式碼編輯能力才能更新。若未來維護者轉換，高門檻將造成資料停滯。

本計畫分兩階段：Phase A 重構資料模型並建立甘特圖展示；Phase B 將資料遷移至 Firebase Firestore，並建立圖形化後台管理介面。
Design Decisions
核心設計決策
遊戲規律：所有更新固定於星期四

鋼嵐的版本更新、活動開始、活動結束均以星期四為單位。因此活動的時間資訊可用「起始日（星期四）+ 持續週數」完整描述。
例：5/21（週四）開始、1週的活動 → 結束於 5/27（週三）。
結束日公式：endDate = startDate + weeks × 7 − 1 天
活動時間線版面：三欄式 + 垂直時間軸

版本展開面板維持三欄式佈局：
左欄 = 陸版活動時間線（CN）、中欄 = 版本核心資訊、右欄 = 台版活動時間線（TW）

左右兩欄的時間軸為縱向（由上而下），每一列 = 一週（週四 → 下週三）。
每個活動佔用一條縱向欄道（track），活動期間的週格填充對應顏色色塊，持續多週則色塊連續填充。
跨越上/下半版本界線的活動，其色塊會在分界線兩側都出現，視覺上一目了然。

核心資訊（機師、機甲、武裝討伐、戰令）不隨活動時間變動，顯示於中欄即可，不需要色塊欄道。
活動類資訊（復刻卡池、限時活動、刮刮樂、輪盤）各佔一條欄道，以週為單位呈現持續時間。
陸台分離資料輸入

同一個活動在陸服與台服的起始日不同（通常差約半年），且台服有時會調整活動順序。
因此甘特圖的活動（TimedActivity）採用「陸服、台服各自輸入」的模式，不從陸服日期自動推算台服日期。
這也是為何版面設計左右兩側獨立，而非共用同一份資料。
Phase A — Data Model
Phase A：TimedActivity 資料模型擴充
重構範圍：PatchHalf 中的活動欄位從 string[] 升級為 TimedActivity[]

目前 specialEvents、revivedBanners、skinGacha、rouletteEvent 均缺乏時間維度。
重構後每筆活動帶有 startDate（起始日）與 weeks（持續週數），UI 可計算結束日並繪製甘特條。
核心欄位（pilots、mechs、armamentRaids、battlePass）保持不變，因為它們以版本半起始日為準，不需獨立時間區間。

遷移策略：漸進式，不破壞現有顯示

舊欄位（skinGacha、rouletteEvent、specialEvents、revivedBanners）保留，UI 優先讀取新的 cnActivities / twActivities；若新欄位不存在則 fallback 顯示舊欄位的文字清單。
這樣可以邊填新資料邊保持現有功能正常，不需要一次性完成所有版本的資料遷移。

三欄式活動時間線概念

現有 VersionExpandedPanel 的陸台雙欄改為三欄：
左欄（陸版活動時間線）、中欄（版本核心資訊）、右欄（台版活動時間線）

左右欄的時間軸為縱向：每一列代表一週（週四到下週三），從各自半版本起始日向下排列。
每個活動各佔一條獨立欄道（track），活動期間的週格填充色塊，持續多週的活動形成連續色條。
跨越上/下半界線的活動（如卡池持續到下半第一週）其色塊在分界線兩側連續延伸，視覺上直觀呈現橫跨版本的事實。
不同 ActivityType 使用不同顏色，方便一眼區分角雕特遣、跨域海運、卡池、刮刮樂、限時活動等。

手機版降格策略

三欄佈局在手機版（< 768px）無法正常顯示。手機版降格為 Tab 切換模式：
Tab 1「核心資訊」—— 中欄 CoreInfoPanel 全寬展示（機師/機甲/武裝/戰令）
Tab 2「陸版活動」—— 陸版 ActivityTimeline 全寬顯示（垂直週列色塊）
Tab 3「台版活動」—— 台版 ActivityTimeline 全寬顯示（同上）

垂直週列結構在手機版全寬顯示不需要更動，只是不再三欄並排。
Phase B — Firebase Admin
Phase B：Firebase Firestore 遷移 + 後台管理 UI
核心目標：讓非技術維護者透過網站內的管理介面更新版本資料，無須接觸程式碼或 TypeScript 語法。

資料從靜態 TypeScript 檔案遷移至 Firebase Firestore。前台讀取改用 Firestore hook；後台管理頁提供完整的表單介面，覆蓋所有欄位（含 TimedActivity 的日期選擇器和週數輸入）。
暫定技術選型：Firebase Storage（圖片）+ Firebase Firestore（資料）

圖片儲存 → Firebase Storage
Admin UI 提供「上傳圖片」按鈕，維護者選擇本機檔案後自動上傳至 Firebase Storage，URL 自動回填 Firestore，不需手動複製路徑。
Storage 安全規則：isAdmin 才可寫入；所有人可讀取（公開 URL）。
免費額度：5 GB 儲存 / 月 1 GB 下載，對此規模完全足夠。

版本資料 → Firebase Firestore
專案已使用 Firebase Auth，同一 project 內加 Firestore 無額外設定；安全規則直接綁定 Auth 的 isAdmin 欄位，不需後端 API。
onSnapshot 即時同步：Admin 存檔後前台立即反映，無需刷頁。
免費額度：每日 5 萬次讀取 / 2 萬次寫入，版本資料讀多寫少完全足夠。

選用理由：統一在 Firebase 生態，最小化新增依賴；若未來有調整需求，Fallback 至靜態 TS 資料的機制已在 Phase A 規劃中預留。
// Firestore 集合結構 patchVersions/ ← Collection（對應 PatchVersion[]） v2.8/ ← Document ID = version 號 version: "2.8" bannerImage: "/images/banners/v2.8.jpg" isTwCurrent: false upper: { ... } ← PatchHalf（含 cnActivities / twActivities） lower: { ... } crisisShop: [...] ... v3.0/ isTwCurrent: true ... // 讀取端：替換靜態 import // Before: import { PATCH_VERSIONS } from '../data/patchVersions' // After: const { data: PATCH_VERSIONS } = usePatchVersions()

Firestore 安全規則

  • 所有人可讀取 patchVersions
  • 寫入需 isAdmin: true 於 users profile
  • Admin 認定：由現有管理員於 Firebase Console 手動設定 users/{uid}/profile/main.isAdmin
  • 無需 Firebase Admin SDK 或 Cloud Functions

usePatchVersions hook 設計

  • 讀取 Firestore patchVersions，依 version 號排序
  • Firestore 讀取失敗時 fallback 至靜態 TS 資料
  • loading / error 狀態管理
  • 快取：onSnapshot 保持即時同步（管理員更新後前台立即反映）

  • 持續週數（數字輸入，min=1，顯示自動計算的結束日）
  • 活動類型（下拉選單：復刻卡池 / 刮刮樂 / 輪盤 / 限時活動 / 簽到活動 / 戰令）
  • 刪除此筆活動按鈕
UX 細節
  • Date picker 旁顯示「星期幾」提示，非週四時顯示紅色警告（遊戲固定週四改版）
  • 週數輸入旁即時顯示計算出的結束日，例如輸入 1 顯示「結束：05/27（週三）」
  • 儲存前顯示摘要確認，防止誤操作
  • 儲存成功後 toast 通知，前台（Firestore onSnapshot）即時更新
  • 版本列表顯示各版本「陸服活動」/ 「台服活動」填寫進度，方便知道哪些版本資料不完整

Maintenance
Phase B 完成後：非技術維護者操作流程
台服版本更新時（每約 3 週）
  1. 打開工具站，以管理員帳號登入
  2. 進入「後台管理」→「版本列表」
  3. 點擊需要更新的版本（如 v3.0）
  4. Tab「上半」或「下半」：修改台服起始日、移除「預測」勾選
  5. 「陸服活動」/ 「台服活動」：新增或修改本次確認的活動資訊（名稱、起始日、週數、類型）
  6. 切換「核心資訊」Tab：若此版本已成為台服當前版本，勾選「★ 台服當前版本」，並取消舊版本的同一勾選
  7. 按「儲存」，網站即時更新
新陸服版本上線時
  1. 後台管理 → 「建立新版本」
  2. 填入版本號（如 3.4）、版本前瞻圖路徑（圖片先上傳至 public/images/banners/）
  3. Tab「上半」：填陸服起始日（精確）、機師/機甲名稱；台服日期填預測值並勾選「預測」
  4. 逐步填入武裝討伐、戰令、各週活動（起始日 + 週數）
  5. 儲存後，版本濃縮資訊表和時間線自動出現此版本
Phase B 的前提條件

Phase B 實作前需先完成 Phase A（TimedActivity 資料模型），因為後台 UI 的 AdminTimedActivityEditor 依賴此型別。
另外，Phase B 需要有至少一個 Firebase 管理員帳號（在 Firebase Console 手動設定 isAdmin: true）才能驗證後台功能。

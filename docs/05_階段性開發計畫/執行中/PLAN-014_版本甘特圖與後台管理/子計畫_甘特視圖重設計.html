背景與動機

PLAN-014 Phase A 實作了縱向週列甘特圖（ActivityTimeline.tsx），以「每一橫列 = 一週」的方式呈現活動期間。 雖然功能完整，但視覺呈現方式與使用者對版本排程的直覺不完全一致：

使用者更習慣「時間由左至右流動」的橫向排版，且希望一眼看到 上半版本、下半版本並排、搭配第 N 週更新時間點作為橫軸， 這樣就能直接對照「某個活動在第幾週開始、持續幾週」。

此外，陸服和台服的活動資料雖獨立維護，但使用者希望能一鍵切換陸版／台版顯示， 而非在兩個獨立面板中左右對比——特別是台服玩家只需要台版資訊時更為直觀。
與 PLAN-014 Phase A 的關係

本子計畫取代 Phase A 的 ActivityTimeline.tsx（縱向）和 VersionExpandedPanel.tsx（三欄式）， 改用新的 VersionGanttPanel.tsx（橫向週欄式）。
資料模型（TimedActivity、cnActivities、twActivities）保持不變， 不需修改型別定義或現有資料。
Phase B 的後台管理 UI 完全不受影響。

Algorithm
週軸計算演算法
核心原則：版本起始日（週四）為第 1 週週四，之後每 7 天為下一週

鋼嵐所有版本更新固定於星期四。 每個「版本週」為週四 00:00 到下週三 23:59。 因此只需版本起始日，即可推算第 N 週的日期。
// 週軸推算（以台版為例） const side = 'tw' // 或 'cn'，由切換狀態決定 // Step 1：取得上半、下半起始日 const upperStart = parseDate(side === 'tw' ? version.upper.twDate : version.upper.cnDate) const lowerStart = parseDate(side === 'tw' ? version.lower.twDate : version.lower.cnDate) // Step 2：計算上半的週列（預設 3 週，可延伸） // upperWeeks = [upperStart, upperStart+7, upperStart+14, ...] // 以 lowerStart 為終點，最少 3 週： function generateWeeks(startDate, endDate, minWeeks = 3) { const minEnd = addDays(startDate, minWeeks * 7) const actualEnd = endDate > minEnd ? endDate : minEnd const weeks = [] let cur = startDate while (cur < actualEnd) { weeks.push(cur) cur = addDays(cur, 7) } return weeks // 每個元素 = 該週的週四日期 } // 上半週：upperStart → lowerStart（自然邊界，min 3 週） const upperWeeks = generateWeeks(upperStart, lowerStart) // 下半週：lowerStart → 最後活動結束日（min 3 週） const acts = version.lower[side === 'tw' ? 'twActivities' : 'cnActivities'] ?? [] const lowerEnd = maxActivityEnd(acts, addDays(lowerStart, 21)) const lowerWeeks = generateWeeks(lowerStart, lowerEnd) // Step 3：週欄標題 // upperWeeks[0] = 第1週（4/23），upperWeeks[1] = 第2週（4/30），依此類推 // lowerWeeks[0] = 第4週（5/14），lowerWeeks[1] = 第5週（5/21） // 整體週序號 = 1 ~ upperWeeks.length + lowerWeeks.length
活動甘特條定位：對應到第幾個欄位

每個 TimedActivity 有 startDate（週四）和 weeks（持續週數）。
活動條的起始欄：在所有週欄中找到 startDate 對應的欄索引。
活動條的欄寬：= weeks 欄（每個週數佔一欄）。

若 startDate 早於版本開始（如復刻卡池從前一版本延續），則欄索引 = 0（從第一欄開始）。
若 startDate 晚於最後一欄的週四，該活動不顯示。
若活動跨越上半/下半邊界（例如 startDate 在上半末週，weeks=2 延伸到下半第一週），條的寬度自然跨越中間的分隔線。
// 活動條欄位計算 function activityToColumns(act, allWeeks) { // allWeeks = [...upperWeeks, ...lowerWeeks]，每個元素是週四日期 const startDt = parseDate(act.startDate) const endDt = addDays(startDt, act.weeks * 7) // 結束週四（不含） // 起始欄索引：找第一個 week >= startDt let colStart = allWeeks.findIndex(w => w >= startDt) if (colStart === -1) return null // 活動在顯示範圍外 // 結束欄索引：找最後一個 week < endDt let colEnd = allWeeks.findLastIndex(w => w < endDt) if (colEnd < colStart) return null return { colStart, colSpan: colEnd - colStart + 1 } } // 使用 CSS grid 或 table colspan 定位甘特條 // colStart + 1 = grid-column-start（1-indexed） // colSpan = grid-column-end span
Architecture
元件架構
目標：用一個 VersionGanttPanel 取代舊的三欄式 VersionExpandedPanel

現有的 ActivityTimeline.tsx（縱向）+ CoreInfoPanel.tsx（中欄）+ 三欄布局
→ 全部整合成 VersionGanttPanel.tsx（橫向週欄式）

VersionDetailView.tsx 和 VersionTimeline.tsx 不需要修改， 只有 VersionExpandedPanel.tsx 改為使用 VersionGanttPanel。
// 元件結構 VersionGanttPanel ← 主元件，接收 version: PatchVersion ├── side state ← 'tw' | 'cn'，預設 'tw' ├── ToggleButton ← 右上角 台版⇄陸版 切換按鈕 ├── ContentTable ← 整個表格（版本資訊行 + 活動甘特行） │ ├── ColHeaders ← 上半/下半標題行 + 第N週欄標行 │ ├── VersionInfoRows ← 機師/機甲/武裝/戰令（靜態跨欄內容） │ └── ActivityGanttRows ← 活動甘特條行（動態，每個活動一行） └── VersionLevelInfo ← 危境重構/邊境商店等版本層級資訊（表格外） // props 設計 interface VersionGanttPanelProps { version: PatchVersion } // side 狀態內部管理，不透傳給父元件

ColHeaders（欄標題列）

  • 第一行：空白 label 欄 ＋「上半版本 + 版本起始日」colspan=3 ＋「下半版本 + 版本起始日」colspan=3
  • 第二行：label 欄 ＋ 第1週(日期) / 第2週(日期) / ... 逐欄
  • 週日期依 side 切換後重新計算（cn/tw 的起始日不同）
  • 欄數 = upperWeeks.length + lowerWeeks.length（動態）

VersionInfoRows（版本內容行）

  • 每行：label ＋ 上半內容（colspan=上半週數）＋ 下半內容（colspan=下半週數）
  • 機師：pilots 列表以逗號分隔，跨上半所有欄
  • 武裝：armamentRaids 各自一行，名稱 + 武器/背包
  • 戰令：跨整個版本（colspan = 上半週數 + 下半週數）
  • 欄位無資料時自動隱藏該行（不顯示空行）

ActivityGanttRows（活動甘特條行）

  • 依 side（cn/tw）選擇 upper.cnActivities/twActivities + lower.cnActivities/twActivities
  • 每個 TimedActivity 各佔一行，label 欄顯示活動名稱（或短代碼）
  • 以 activityToColumns() 計算 colStart + colSpan，用 CSS grid 或 table 定位彩色甘特條
  • 同顏色系統：ActivityType → COLORS map（與 Phase A 相同）
  • 甘特條 hover tooltip 顯示：活動名稱 + 起訖日（自動計算）
  • 跨越上半/下半邊界的活動自然橫跨分隔線（colspan 涵蓋兩側欄位）
實作選擇：HTML <table> vs CSS Grid

推薦使用 HTML <table>：因為 colspan 的跨欄邏輯天然符合表格結構， 且「欄寬等比」在 table-layout: fixed 下較易控制。

CSS Grid 方案：使用 grid-column: colStart / span colSpan 可以達到相同效果， 但需要為每個格子明確設定 grid-column，程式碼較為繁瑣。

建議以 HTML table 為主，Tailwind 的 CSS 仍可正常套用； 若未來需要拖曳排序活動列，再評估改用 Grid。
Mobile
手機版策略
橫向寬表格在手機版需要橫向捲動

6 欄（或更多）的週欄表格在 375px 寬的手機上無法完整顯示。
解決方案：將整個 VersionGanttPanel 包裹在 overflow-x: auto 的容器中， 允許使用者左右滑動查看完整週欄。

最小欄寬設定：每週欄至少 64px，label 欄至少 80px，確保文字不被壓縮到無法閱讀。
整個表格最小寬度 = 80 + 64 × (週欄數)，在手機上觸發橫向捲動。

此策略比 Tab 切換更直觀：使用者可以直接拖動，不需在三個 Tab 之間切換上下文。

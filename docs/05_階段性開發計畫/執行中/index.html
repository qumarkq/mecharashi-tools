PLAN-019 · 實體引用與模擬基礎（總綱）

【總綱 Master Plan】把「描述文字中對其他實體的引用」（如天賦內 [虛粒子形態]/[激發能]/[粒子爆發]）一般化為型別化、可解析、可模擬的關聯層。四層抽象：EntityRef 引用層、BUFF/形態模擬可達性圖、BOSS/關卡資料模型、表單式後台 + 查詢量控制。統合 PLAN-001/003/004/005 為子計畫，並拆出 019-A~E。

PLAN-010 · 攻略系統

以 TipTap WYSIWYG 編輯器建立攻略 CMS，支援圖文並茂內容與 Firebase Storage 圖片上傳（Phase A）。Phase B 擴充自訂 EntityReferenceNode，可嵌入遊戲資料庫物件（機師/機甲/武器卡片），前台局部 Drawer 展開詳情，不跳頁。管理員寫、玩家讀。

PLAN-016 · Firebase 快取層

建立 GameDataContext 全局快取層，App 啟動時一次讀取全部 11 個 game data collection，頁面切換完全走 in-memory。同步修正 usePilotExclusiveWeapon/s、useMechWithModules 的全量讀取查詢，以及 usePatchVersions 的 real-time listener。估計降低 70–80% Firestore 讀取次數，解決單日超出免費配額問題。

→ 計畫書 → 進度表
◆

PLAN-017 · localStorage 版本快取

在 PLAN-016 的基礎上，讓回訪用戶的遊戲資料完全從 localStorage 讀取，每次 session 僅消耗 1 次 Firestore 讀取（版本比對）。Firestore 新增 meta/gameData 版本文件，GameDataContext 啟動時先讀版本，命中快取則跳過 11 次讀取。Admin 一鍵 Bump 版本號即可使全站快取失效。估計回訪用戶讀取降幅 91%。

→ 計畫書 → 進度表

PLAN-014 · 版本甘特圖與後台管理

Phase A：為版本展開面板加入甘特圖展示，活動（復刻卡池、刮刮樂、輪盤等）以起始日＋持續週數繪製橫條，陸台各自獨立輸入，三欄式版面（陸版甘特圖｜核心資訊｜台版甘特圖）。Phase B：版本資料遷移至 Firebase Firestore，建立圖形化後台管理介面，讓非技術維護者無須碰程式碼即可更新版本內容。

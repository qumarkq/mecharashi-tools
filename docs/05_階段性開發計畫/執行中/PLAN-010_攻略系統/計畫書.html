攻略系統Guide CMS · TipTap · Entity Integration

以 TipTap WYSIWYG 編輯器為核心，支援圖文並茂的攻略撰寫，並可嵌入遊戲資料庫物件（機師/機甲/武器卡片），前台局部預覽不跳頁

Requirements
需求定義
核心需求
  • 管理員可透過 WYSIWYG 編輯器撰寫圖文並茂的攻略，圖片上傳至 Firebase Storage
  • 攻略內容可嵌入遊戲資料庫物件（機師、機甲、武器、模組等），嵌入後顯示對應卡片
  • 前台玩家點擊嵌入卡片可局部展開詳情（Drawer/側滑面板），不跳頁、不離開攻略
  • 管理員寫、玩家讀；草稿 / 發布狀態分離
  • 安全層：只有 admin custom claim 持有者可寫入
⚠️ 複雜度說明
本計畫分為兩個 Phase 開發。Phase A 為基礎 CMS（TipTap + 圖片 + Firestore），難度中等，可獨立上線。 Phase B 為遊戲資料物件嵌入，需自訂 TipTap Extension + React NodeView，難度中高； 若評估後認為工期不符合效益，可改用備選方案（見本文件末段）。
Architecture
整體架構設計

內容儲存

  • Firestore guides/{guideId} — 文章 metadata + 內文 JSON
  • 內文格式：TipTap ProseMirror JSON（非 HTML），完整保留自訂節點資訊
  • Firebase Storage guides/images/{guideId}/ — 所有圖片

編輯器（後台）

  • TipTap WYSIWYG，方案 B
  • 自訂 EntityReferenceExtension 產生物件節點
  • 物件節點以 React NodeView 渲染，編輯器內即可預覽卡片
  • 圖片工具列按鈕 → Storage 上傳 → 取得 URL → 插入

前台展示

  • 以 TipTap editable: false 模式渲染相同 JSON
  • 同一套 Extensions → 自動渲染物件節點為互動卡片
  • 點擊卡片 → 右側 Drawer 局部展開詳情（複用現有詳情頁元件）
  • 玩家全程停留在攻略頁

安全層

  • 路由守衛：admin custom claim 驗證，未授權重導首頁
  • Firestore Rules：guides 寫入限 admin；已發布文章公開讀取
  • Storage Rules：圖片上傳限 admin；讀取公開
為何使用 ProseMirror JSON 而非 HTML？
TipTap 自訂節點（EntityReferenceNode）含有 entityType 與 entityId 等 attrs， 這些資訊在序列化成 HTML 時會遺失（HTML 無法可靠攜帶結構化 attrs）。 JSON 格式原生保留完整節點樹，前台用同一組 Extensions 反序列化後自動渲染，不需額外 parser。

Phase B
Phase B — 遊戲資料物件嵌入（進階，難度中高）

讓攻略內容可嵌入遊戲資料庫物件（機師卡/機甲卡/武器卡等），嵌入節點在編輯器與前台均以 React 元件渲染，前台點擊可局部展開詳情。

核心機制：TipTap 自訂 Node + React NodeView
TipTap 支援將任意節點的渲染委託給 React 元件（ReactNodeViewRenderer）。 自訂的 EntityReferenceNode 僅儲存 entityType + entityId + displayMode， 渲染邏輯由 React 元件完全控制，編輯器與前台共用同一套元件，確保所見即所得。
// EntityReferenceNode — TipTap 自訂節點 attrs（ProseMirror JSON 內容） { "type": "entityReference", "attrs": { "entityType": "pilot" // "pilot" | "mech" | "weapon" | "module" | "backpack" | "component", "entityId": "pilot_kairi", "displayMode": "card" // "card"（完整卡片）| "badge"（行內小標籤） } }
// EntityReferenceExtension.ts（概念骨架） import { Node, ReactNodeViewRenderer } from '@tiptap/react' import { EntityCard } from '../components/EntityCard' export const EntityReferenceExtension = Node.create({ name: 'entityReference', group: 'block', atom: true, // 不可分割的原子節點 selectable: true, addAttributes() { return { entityType: { default: null }, entityId: { default: null }, displayMode: { default: 'card' }, } }, addNodeView() { return ReactNodeViewRenderer(EntityCard) }, })
工作項目 說明 難度 估時
B-1 EntityReferenceExtension 自訂 TipTap Node；addAttributes、parseHTML/renderHTML（JSON 模式此處可簡化）；ReactNodeViewRenderer 掛載 中 1–1.5 天
B-2 EntityCard 元件 依 entityType 分派渲染對應卡片（PilotMiniCard / MechMiniCard / WeaponMiniCard 等）；在編輯器內自 Firestore 讀取資料並顯示；含 loading / error 狀態 中 1.5–2 天
B-3 後台物件搜尋器 工具列「插入物件」按鈕 → Popover 搜尋框（按名稱搜尋）→ 顯示搜尋結果列表 → 點擊插入對應節點 中高 1.5–2 天
B-4 前台批次預載 前台渲染攻略前，解析 JSON 取出所有 entityReference 節點，依 entityType 分組批次 Firestore 查詢，存入 Context 供 EntityCard 使用（避免 N+1） 中 1 天
B-5 EntityDetailDrawer 右側滑入 Drawer 元件；複用現有 PilotDetailPage / MechDetailPage / WeaponDetailPage 的詳情區塊（非整頁，僅核心資訊）；點擊卡片觸發，按 ESC 或點擊遮罩關閉 中 1–1.5 天
Phase B 合計估時：約 6–8 天（+1.5–2 週）
⚠️ Phase B 主要風險點
  • B-2 EntityCard 維護成本：每新增一種資料類型（如 backpack）都需補寫對應的 MiniCard，需要長期維護
  • B-3 物件搜尋器 UX：多種資料類型同時搜尋的 UI 設計需要細心規劃，否則編輯體驗差
  • B-4 前台批次載入：若一篇攻略嵌入多種資料，批次邏輯需要妥善處理，避免初始渲染過慢
  • TipTap 版本鎖定：ReactNodeViewRenderer API 在 TipTap v2 與未來版本間可能有 breaking change
Fallback
備選方案 — Mention 輕量版（Phase B 的替代選項）
適用時機：Phase B 工期估算後認為不划算，或初期攻略數量少、不需要深度整合時

做法：使用 TipTap 官方 @tiptap/extension-mention， 在攻略內以 @凱莉 的方式引用遊戲物件。 Mention 節點存 id 與 label，前台渲染為帶顏色的連結標籤（badge）。 點擊後直接跳轉到對應詳情頁（現有路由），不需要 Drawer。

工期：約 2–3 天（Phase A 完成後即可追加）
缺點：玩家閱讀攻略中途需要跳頁，閱讀體驗中斷；不支援卡片式預覽

// Firestore Security Rules（新增段落） match /guides/{guideId} { // 玩家：只能讀已發布的文章 allow read: if resource.data.status == 'published'; // 管理員：完整讀寫（含草稿） allow read, write: if request.auth.token.admin == true; } // Firebase Storage Rules match /guides/images/{allPaths=**} { allow read: if true; allow write: if request.auth.token.admin == true; }

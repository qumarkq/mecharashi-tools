Problem
問題背景
現有武器 Schema 在射程、限制、術語、稀有度上的設計缺陷。
問題一：射程欄位為字串，無法結構化處理
現有 range: string 以 "2-4"、"1+" 等字串儲存射程，程式端需要額外解析才能比對數值。 此外，遊戲中射程有兩種型態：線性射程（如 "2-4"，有最小射程限制）與 環形 N 圈（如 "1+"、"2+"，以持有者為中心的 (2N+1)×(2N+1) 方形區域），兩者語義不同， 但現有欄位無法區分。計算器若需要判斷「敵人在射程內」或「模組條件：近距離」，必須依賴字串解析，容易出錯。
問題二：缺少機甲限制欄位
部分武器有使用限制，只能裝備於特定裝甲類型的機甲（輕型 / 中甲 / 重型）。 目前 Weapon interface 沒有此欄位，無法在前端篩選或計算器中驗證裝備合法性。 相比之下，Backpack interface 已有 mechRestriction 欄位可作為參照。
問題三：類型 / 種類術語混淆，欄位命名與中文概念倒置
遊戲中武器分類有兩個層級：
· 類型（大分類）：射擊 / 格鬥 / 突擊 / 戰術
· 種類（細分類）：機槍 / 狙擊步槍 / 霰彈 / 刀劍 / 拳套…

但現有 interface 中 category（WeaponCategory）對應「類型」，而 type 對應「種類」， 英文欄位名與中文概念恰好相反。「種類」未來是模組生效條件的關鍵篩選維度，若欄位名稱混淆， 計算邏輯開發時將產生大量認知負擔與潛在錯誤。
問題四：稀有度 Enum 不符合武器實際分級
現有通用 ItemRarity（EX / S / A / B）的分級邏輯與武器稀有度無關。 武器稀有度僅三級，且識別規則完全不同：
· SS：專屬武器（特殊名稱，最高等級）
· S+：名稱含「改」（強化版本）
· S：名稱含「EX」

目前 rarity: string 為無型別的字串，既無法靜態驗證，也無法在 UI 元件中直接對應樣式。
問題六：武器種類（kind）缺乏列舉型別保護
kind: string 為無型別字串，填值無靜態驗證，也無法保證與模組 weaponKind? 條件使用一致的字串。 遊戲中武器種類固定（共 17 種），應建立 enum 明確定義，讓 TypeScript 在編譯期驗證填值合法性。
問題七：武器缺少裝備部位欄位；機甲缺少肩膀 / 背後武器槽資訊
遊戲中武器有四種裝備部位：單手（左臂或右臂）、雙手（同時佔據左右臂）、 肩膀（左臂或右臂的肩膀槽）、背後（背後槽）。
目前 Weapon interface 沒有裝備部位欄位，計算器無法驗證裝備合法性（如雙手武器佔用雙臂）。
同時，Mech interface 沒有記錄機甲是否擁有肩膀 / 背後武器槽， 以及少部分機甲將特殊部件綁死於肩膀的情況，導致計算器無法驗證肩膀 / 背後武器的裝備合法性。
問題五：專武被動效果填值模式不明確
專屬武器（isExclusive: true）有類似模組的條件式被動效果，例如「對輕型機甲攻擊時傷害 +X%」。 這些效果在 PLAN-001 完成後，技術上可透過 skills[].activation = 'carry' 搭配 SkillEffect[] 表達， 但目前文件和填值指引未說明此模式，導致管理員容易漏填或只填文字描述而不填結構化數值。
Design
設計決策
針對上述問題的解決方案與設計取捨說明。
決策一：射程拆分為三個結構化欄位
移除 range: string，改以三個欄位表達：
· rangeType: RangeType — 'linear'（線性，有最小射程限制）/ 'ring'（環形 N 圈，以持有者為中心的方形區域）
· minRange: number — 最小射程。linear 型填遊戲的下限值（如狙擊 2-4 → 2）；ring 型固定填 0（無最小限制）
· maxRange: number — 最大射程（如狙擊 2-4 → 4；2+ → 2）

計算器及篩選器可直接做數值比較，不再需要解析字串。
決策二：直接採用既有 MechRestriction enum
MechRestriction（none / light / medium / heavy）已存在於 enums.ts，無需新增。 Weapon interface 新增 mechRestriction: MechRestriction 欄位，無限制的武器填 'none'。 此設計與 Backpack 的處理方式一致（Backpack 使用 string | null，武器改用 enum 更嚴格）。
決策三：欄位重命名以對齊中文術語
為消除類型 / 種類術語混淆，進行以下重命名：
· Enum：WeaponCategory → WeaponType（對應「類型」：射擊 / 格鬥 / 突擊 / 戰術）
· 欄位：category → type（使用 WeaponType enum）
· 欄位：type → kind（機槍 / 狙擊步槍 / 刀劍…，plain string）
· 欄位：typeCoefficient → kindCoefficient（跟隨種類重命名）

同步更新 PLAN-001 定義的 SkillCondition：weaponCategory? → weaponType?， 並新增 weaponKind? 以支援種類層級的模組條件篩選。
決策四：新增 WeaponRarity enum，取代無型別字串
建立武器專屬稀有度 enum：SS / S_PLUS / S，對應中文「SS / S+ / S」。 稀有度識別規則（根據武器名稱判斷）記錄於文件與 weapons.html 說明表， 不在 Schema 中自動推斷，管理員填資料時手動判斷並填入正確 enum 值。
決策五：專武被動效果 = skills[] 中 activation='carry' 的技能
不新增額外欄位。專武的條件式被動效果以 skills[] 中一筆 activation: 'carry' 的 WeaponSkill 表達，其 effects: SkillEffect[] 依模組效果格式填入結構化數值。

計算器處理武器 carry 技能的邏輯與模組效果完全相同，不需要額外的處理分支。 文件補充填值範例與說明，確保管理員理解此模式。
決策六：建立 WeaponKind enum，取代無型別 kind string
定義 17 個值的 WeaponKind enum（格鬥類 7 種、射擊類 10 種），與 模組 SkillCondition 中 weaponKind? 欄位共用同一組字串值，確保填值一致性。
kind: string 欄位在 interface 中保留為 string（與其他 enum 欄位慣例一致）， 填值時參照 WeaponKind enum 值，TypeScript 編譯期靜態驗證由管理員工具層處理。
決策七：新增 WeaponEquipSlot enum；機甲新增肩膀 / 背後武器欄位
建立 WeaponEquipSlot enum：singleHand / dualHand / shoulder / back。
Weapon interface 新增 equipSlot: WeaponEquipSlot，標記武器佔用的裝備格位。

Mech interface 新增三個選填欄位：
· leftShoulderSlot?: string | null — undefined = 無左肩槽；null = 空槽；string = 固定武器 ID
· rightShoulderSlot?: string | null — 同左肩邏輯
· backSlot?: string | null — 背後武器槽，邏輯相同

此設計讓計算器可直接透過 equipSlot 知道武器需要哪種槽位， 再查詢目標機甲是否有對應槽位，做出裝備合法性判斷。

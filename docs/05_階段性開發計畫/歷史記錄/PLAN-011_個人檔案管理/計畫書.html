Requirements
需求定義
核心需求
  • 用戶可在個人中心修改網站顯示名稱（displayName）
  • 用戶可自訂頭像：上傳圖片（→ Firebase Storage）或從機師庫中選取機師頭像
  • 用戶可選填遊戲暱稱（gameNickname，可空白）
  • 用戶可選填遊戲伺服器（gameServer，enum，可空白）：獵豹物流 / 灰燼之子 / 破曉世紀
  • 用戶可選填遊戲公會（guild，可空白）
  • 以上所有欄位均為選填，不影響核心功能使用
⚠️ 頭像設計決策
頭像來源分為兩種模式，以 avatarType 欄位區分：
  • upload：用戶自行上傳圖片，壓縮為 WebP 後存入 Firebase Storage，URL 存於 avatarUrl
  • pilot：從機師庫選取機師，儲存機師 ID（avatarPilotId），顯示時即時取得機師頭像圖片
未設定頭像時（avatarType: null），顯示預設 Avatar（以顯示名稱首字母產生）。

Firebase Storage — 頭像路徑
// 上傳頭像的 Storage 路徑（每次上傳覆蓋同一個檔案） avatars/{uid}/avatar.webp // Firestore Security Rules 新增段落 match /avatars/{uid}/{allPaths=**} { // 頭像圖片公開可讀（用於顯示） allow read: if true; // 只有本人可寫入自己的頭像，限制 2MB + 圖片格式 allow write: if request.auth.uid == uid && request.resource.size < 2 * 1024 * 1024 && request.resource.contentType.matches('image/.*'); }
為何上傳前壓縮為 WebP？
WebP 相較 PNG/JPG 體積約小 25–35%，節省 Storage 費用與讀取頻寬。 壓縮在前端進行（使用 canvas.toBlob('image/webp', 0.85)），不需後端。 舊版 Safari（iOS 13 以下）不支援 WebP，但本專案目標平台可接受此限制。
Architecture
整體架構設計

資料層

  • Firestore users/{uid}/profile — 新增 7 個欄位
  • Firebase Storage avatars/{uid}/avatar.webp — 上傳頭像
  • src/lib/profileApi.ts — 封裝讀寫邏輯

頭像解析邏輯

  • avatarType = null → 顯示名稱首字母預設 Avatar
  • avatarType = 'upload' → 顯示 avatarUrl（Storage 圖）
  • avatarType = 'pilot' → 從 pilots collection 取 avatarPilotId 對應的機師頭像

UI — 頭像選取器

  • 兩個分頁：「上傳圖片」/ 「機師頭像」
  • 上傳頁：拖放區 + 按鈕、預覽、確認上傳
  • 機師頁：搜尋欄 + 格狀機師列表、點擊選取、高亮已選
  • 以 Modal 或 Drawer 呈現（非跳頁）

UI — 個人資料表單

  • 位於 /profile 頁的「個人資料」Tab
  • 顯示名稱：text input
  • 遊戲暱稱：text input（選填提示）
  • 遊戲伺服器：select（含「未設定」空值選項）
  • 遊戲公會：text input（選填提示）
  • 儲存按鈕：單次提交至 Firestore

Notes
注意事項 / 邊界條件
⚠️ 機師頭像讀取效能
AvatarPicker 機師 Tab 需讀取所有機師列表。若機師數量多，應使用現有的 usePilots hook（已有快取）或一次性讀取靜態列表，避免每次開啟 Modal 都發出 Firestore 請求。
⚠️ avatarType='pilot' 的頭像來源
機師頭像的實際圖片 URL 需在顯示時動態取得（從 pilots collection 讀取），不儲存靜態 URL 至 profile。 這樣管理員更新機師頭像後，所有選用該機師頭像的用戶會自動同步最新圖片。
⚠️ 顯示名稱長度限制
建議前端限制 displayName 最長 30 字元，gameNickname / guild 最長 30 字元， 避免 UI 溢出。Firestore Rules 端可選擇性加驗，但非強制。

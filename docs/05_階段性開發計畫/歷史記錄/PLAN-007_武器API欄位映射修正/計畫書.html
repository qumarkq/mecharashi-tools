PLAN-003
發現背景
在為擷取腳本新增 SS/S+ 武器技能自動填入功能（PLAN-003 後續工作）時，實際比對 Chrome DevTools 的 Network 請求發現：API 對 weapon_data detail 端點的確有回傳技能資料， 但擷取腳本一直以來查找的欄位名稱與 API 實際欄位名稱不符，導致技能（及多個基礎屬性）全部漏抓。 此計畫旨在系統性修正所有已知欄位名稱映射錯誤。
Problem
問題背景
透過 Chrome DevTools 直接比對 API 原始回應，發現以下欄位名稱錯誤。
API 實際回應範例（weapon_data detail，武器：爝火）
// GET https://ma-activity.zlongame.com/common/infodata/mQuery.do // ?appkey=1616148215678&target=weapon_data&type=detail&query=爝火 { "ID": "20315121", "name": "爝火", "quality": "SSSR", "WeaponType1": "Assault", "WeaponType2": "ShotGun", "LimitedModelOfWeapon": "Light/Medium/Heavy", // 機甲限制 "RestrictionsPositionOfWeapon": "Hand", // 裝備部位 "range": "1-2", "WeaponWeight": "450", // 重量 "WeaponBasicAttackingPower": "1663", // 基礎攻擊力（總計） "WeaponHitPoint": "3182", // 命中 "WeaponUnderstanding": "4603", // 暴擊值 "PassiveSkill": [ // 武器技能陣列 { "ID": "53035", "name": "阔步", "SpecificEffects": "主动攻击对战前，若移动大于等于<color=#F74848>4</color>格...", "BufCarried": "530351", "SkillIcon": "Icon_skill_passive_5161", "icon": "Icon_skill_passive_5161" } // ... 更多技能 ] }
問題一：武器技能（PassiveSkill）完全未擷取
腳本 buildWeaponSkills() 查找 Skills / skills / WeaponSkills 三個欄位名， 但 API 實際欄位為 PassiveSkill。導致所有武器（含 SS/S+）的技能陣列永遠為空， 技能資料需要管理員全部手動輸入，無法自動化。
問題二：命中（accuracy）一直為 0
腳本查找 Accuracy / accuracy，但 API 欄位為 WeaponHitPoint。 Firebase 中所有武器的 accuracy 欄位因此均為 0（預設值），為錯誤資料。
問題三：暴擊值（critValue）一直為 0
腳本查找 CritValue / critValue / Crit，但 API 欄位為 WeaponUnderstanding。 Firebase 中所有武器的 critValue 均為 0，為錯誤資料。
問題四：重量（weight）一直為 0
腳本查找 Weight / weight，但 API 欄位為 WeaponWeight。 Firebase 中所有武器的 weight 均為 0，為錯誤資料。
問題五：裝備部位（equipSlot）永遠從 KIND_DEFAULTS 取值
腳本查找 EquipSlot / equipSlot / WeaponSlot，但 API 欄位為 RestrictionsPositionOfWeapon（值如 "Hand"、"DualHand"、"Shoulder"、"Back"）。 雖然 KIND_DEFAULTS 的預設值對多數武器正確，但有例外的武器無法被正確修正。
問題六：機甲限制（mechRestriction）永遠從 KIND_DEFAULTS 取值
腳本查找 MechRestriction / mechRestriction，但 API 欄位為 LimitedModelOfWeapon（值如 "Light"、"Medium"、"Heavy"， 或斜線分隔的組合如 "Light/Medium/Heavy" 表示無限制）。 種類預設值（戰術類全部為 medium）對多數武器正確，但無法處理個別武器有不同限制的情況。
問題七：型別定義缺少 API 欄位對照注解
src/types/index.ts 的 Weapon interface 與 src/types/enums.ts 的 WeaponEquipSlot、MechRestriction 等 enum， 均未標注與 API 欄位名稱的對應關係，維護者難以在不查官方 API 的情況下理解欄位來源。
Design
設計決策
決策一：TypeScript 內部欄位名稱不隨 API 命名更改，改以注解標注對應關係
accuracy、critValue、weight 等欄位語意清晰， 若改為 hitPoint（WeaponHitPoint）或 understanding（WeaponUnderstanding） 反而降低可讀性（API 命名為內部代碼，非語意名稱）。
做法：在 interface 欄位上加 // API: WeaponHitPoint 形式的注解， enum 值也加注 API 對應字串，供腳本維護者查閱。
決策二：LimitedModelOfWeapon 斜線組合值 → 以包含的類型數量判斷是否有限制
"Light/Medium/Heavy"（三種都含）→ mechRestriction: 'none'
"Light" / "Medium" / "Heavy"（單一）→ 對應 enum 值
"Light/Medium" 等組合（兩種）→ 標記 __REVIEW__ 待人工確認（目前遊戲中未見此情況）
決策三：PassiveSkill 技能的 activation 欄位以技能類型推斷，預設 'carry'
API 不提供 activation 欄位。武器被動技能性質上屬於「攜帶即生效」， 預設填入 'carry'。管理員可在 Firebase 針對特殊技能手動修改為 'equip' 或 'use'。
腳本以 prev?.skills != null 保護手動填入資料，不會覆蓋已有值。
決策四：WeaponBasicAttackingPower 為總攻擊力，暫不改動 attack 欄位的計算邏輯
API 的 WeaponBasicAttackingPower 是總攻擊力（如 1663），而遊戲顯示為 139×12 （單次攻擊力 × 連擊數）。現行腳本的 attack 欄位來自 list 端點的 MinAttack/MaxAttack， detail 端點提供的是總值，兩者格式不同。
本計畫暫不修改 attack 欄位邏輯，僅補充欄位對照注解，留待後續確認 list 端點欄位格式。
Spec
API 欄位映射規格
腳本修正後，detail 端點欄位與 Firestore/TypeScript 欄位的對應關係。
API 欄位（detail 端點） Firestore / TS 欄位 備注
PassiveSkill[] skills: WeaponSkill[] 技能陣列；管理員已填入時不覆蓋（prev?.skills != null 保護）
WeaponHitPoint accuracy: number 命中值；parseInt 轉換
WeaponUnderstanding critValue: number 暴擊值；parseInt 轉換
WeaponWeight weight: number 重量；parseInt 轉換
RestrictionsPositionOfWeapon equipSlot: WeaponEquipSlot Hand→singleHand / DualHand→dualHand / Shoulder→shoulder / Back→back；API優先於KIND_DEFAULTS
LimitedModelOfWeapon mechRestriction: MechRestriction Light/Medium/Heavy（三含）→none；單一值→對應enum；組合→__REVIEW__
PassiveSkill[].BufCarried skills[].buffIds: string[] 以 "/" 分割，filter(Boolean)
PassiveSkill[].SkillIcon / icon skills[].icon: string key 格式，如 "Icon_skill_passive_5161"；iconLocal 自動補填
Annotation
Enum / Interface 注解補充規格
在現有 TypeScript 定義上加上 API 欄位對照注解，不重命名欄位或 enum key。
// src/types/enums.ts — WeaponEquipSlot 加注 API 對應值 export const WeaponEquipSlot = { SINGLE_HAND: 'singleHand', // API RestrictionsPositionOfWeapon: "Hand" DUAL_HAND: 'dualHand', // API RestrictionsPositionOfWeapon: "DualHand" SHOULDER: 'shoulder', // API RestrictionsPositionOfWeapon: "Shoulder" BACK: 'back', // API RestrictionsPositionOfWeapon: "Back" } as const; // src/types/enums.ts — MechRestriction 加注 API 對應值 export const MechRestriction = { NONE: 'none', // API LimitedModelOfWeapon: "Light/Medium/Heavy"（三種均含） LIGHT_ONLY: 'light', // API LimitedModelOfWeapon: "Light" MEDIUM_ONLY: 'medium', // API LimitedModelOfWeapon: "Medium" HEAVY_ONLY: 'heavy', // API LimitedModelOfWeapon: "Heavy" } as const; // src/types/index.ts — Weapon interface 基礎屬性加注 API 欄位名 export interface Weapon { accuracy: number // API: WeaponHitPoint（命中） critValue: number // API: WeaponUnderstanding（暴擊值） weight: number // API: WeaponWeight（重量） equipSlot: string // API: RestrictionsPositionOfWeapon → WeaponEquipSlot enum // ... mechRestriction API: LimitedModelOfWeapon → MechRestriction enum skills: WeaponSkill[] // API: PassiveSkill[] }
Scope
執行範圍
Phase A — 擷取腳本修正（scripts/scrape-weapons.js）
修正 6 個 API 欄位名稱映射；補充 LimitedModelOfWeapon 斜線組合解析； 補充 RestrictionsPositionOfWeapon 值映射；修正 buildWeaponSkills()。
Phase B — TypeScript 型別注解補充（src/types/enums.ts、src/types/index.ts）
為 6 個受影響欄位加上 API 欄位對照注解；不重命名任何欄位或 enum key；不變更型別定義本身。
Phase C — 技術文件更新（docs/02_技術文件/02_資料模型/weapons.html）
補充技能欄位（WeaponSkill）說明；新增「API 欄位對照表」section； 補充 BufCarried / SkillIcon 子欄位說明。
Phase D — Firebase 資料重刷
執行修正後的腳本全量重刷武器資料（--auto --no-images）； 確認命中 / 暴擊 / 重量 / 機甲限制 / 裝備部位 / 技能全數正確寫入。 技能 icon 另外執行一次含圖示下載的完整更新。
